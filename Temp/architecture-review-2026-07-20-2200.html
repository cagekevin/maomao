二 · 问题分级（全景 · 6 项）

级别问题位置根因 / 影响
P1 ctrlHeld 无失焦重置
（用户报告的"偶尔变框选"）		 L37033–37039 只监听 keydown/keyup，0 处 window.blur/visibilitychange。Cmd+Tab 切走后 keyup 丢失 → ctrlHeld 卡 true → 左键拖拽变框选。这是根因。
P1 ctrl 键位与 RF 内置多选键冲突 L37075–37120 未设键码 未设 selectionKeyCode/multiSelectionKeyCode/panActivationKeyCode，全走默认——Mac 默认 multiSelectionKeyCode=Meta(Cmd)，与自建的 ctrl 翻转 selectionOnDrag 叠在同一键，行为不可预期。设计层隐患。
P2 右键在两模式下语义漂移 L37120 + Cn L31934 无 ctrl：右键=菜单；按 ctrl：panOnDrag:[2] 让右键变平移，菜单是否弹取决于是否移动 → 按 ctrl 想右键菜单时可能拿到平移。
P2 连线落空的 ghost 节点依赖菜单关闭时机 Fr L36195 + effect L36573 落空后 50ms 建 ghost-target+ghost-edge+连线菜单；清理靠"菜单非 connection 类型"的 effect。菜单状态异常时 ghost 边/节点可能残留。
P3 滚轮缩放无自定义 未配 zoomOnScroll/panOnScroll 全默认。Mac 触摸板双指默认缩放而非平移，笔记本用户可能觉得"滚不动画布"。非 bug，属体验取向。
P3 双击边=删除，无二次确认/undo 提示 Yr L36570 误双击静默删连线（是否有全局 undo 需另查）。

三 · 根因归纳

所有交互模式的"分叉"都压在单个布尔 ctrlHeld 上（L31402 状态 + L37034 键监听 + L37119/37120 props），而 ReactFlow 本身已有一整套键码体系（selectionKeyCode/panActivationKeyCode/multiSelectionKeyCode）完全没被使用。等于自建了一套与框架内置机制打架、且缺失失焦兜底的操作模式——两个 P1 都是它的直接后果。这就是本 Module 的"浅"所在：交互模式逻辑摊薄在 3 处，与约 40 个渲染 props 纠缠在同一对象里。

四 · ROI 面板（基于源码事实）

收益 Benefit

一次消除 2 个 P1（失焦卡死 + 键位冲突）+ 缓解 2 个 P2。当前交互模式散落 3 处、与框架键码冲突；收敛为独立 Module 自管重置后，根因消失，交互语义可预期。

成本 Cost

抽 1 个交互模式 Module / Hook，约 新增 30–50 行，改 App.js 3 个位点（L31402 / L37033 / L37119-37120）。无新增依赖，无跨进程调用。最小修复（仅补失焦重置）只需 +3 行。

风险 Risk

RF 默认 selectionKeyCode='Shift'、panActivationKeyCode='Space' 仍生效。若新 Module 不显式声明键码，Shift/空格行为可能与 ctrl 模式叠加。需在 Interface 层显式固定全部 4 个键码避免回归。

六 · 依赖分类（State 2 预判）

全部 14 个鼠标 handler 均为 In-process（纯内存态 + 浏览器事件监听，无 I/O、无网络）。属「永远可深化」类别：直接合并/抽取 Module，透过新 Interface 在组件内测试即可，无需 Adapter。低成本高确定性。

Top recommendation：分两步——①（治标 · P1-1）先给 window 加 blur+visibilitychange → setCtrlHeld(false)，最小 3 行解你当前的"偶尔变框选"；②（治本 · P1-2）去掉自建 ctrlHeld，改用 RF 原生键码把 ctrlHeld+键监听+失焦重置收敛为 1 个 InteractionMode Module，Interface 暴露 { panOnDrag, selectionOnDrag, selectionKeyCode, panActivationKeyCode } 四个已固定键码，同时消除键位冲突。属 In-process 深化，无 Adapter、无跨文件风险，ROI 为正。
