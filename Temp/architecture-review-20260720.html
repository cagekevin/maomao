架构摩擦候选

① localTool 持久化 seam 缺失（运行时数据易失）

Strong

Files：localTool/src/db/database.ts · localTool/src/routes/*.ts

Problem：saveDb() 仅在 closeDb()（SIGINT/SIGTERM 优雅退出）时调用。所有 KV / resources / tasks 的写入只落在 sql.js WASM 内存，进程崩溃、OOM、被强杀即数据全丢。持久化逻辑藏在退出路径，不在写入路径——Locality 被破坏：数据安全这个关键不变量分散在"退出"而非"写入"。

Solution：在 run() / upsertX 写操作后调用 saveDb()（可节流，如每 N 次或每 3s 落盘一次），并把"落盘"做成 db 层内部职责，调用方无感。

Benefits：Locality 收敛——持久化不变量集中在 db 模块；Leverage——上层路由无需关心何时落盘；可测试性提升（可注入"立即落盘"模式做单测）。

② 端点 host 硬编码，绕过 config seam

Worth exploring

Files：localTool/src/routes/resources.ts:30 · src/_engine/config.js · App.js U_/vv

Problem：rescan 入库用 LOCAL_TOOL_BASE='http://127.0.0.1:18080' 本地常量，而非读 config.js 的 LOCAL_ENGINE。App.js 侧 U_/vv 也是硬编码残留。形成"两处真相"——改端口/跨设备需同步改多处，单一配置 seam 被破坏。

Solution：localTool 自身不需要知道前端 host（它是被调用方）；但 App.js 内所有 127.0.0.1:18080 字面量应统一走 config.js 的 LOCAL_ENGINE.base，删掉 U_/vv 等残留硬编码。

Benefits：Locality——端点真相集中；避免 HANDOFF4 那类"相对路径 vs 绝对路径"契约错配复发。

③ 任务记录双份：gateway 内存 vs localTool 表

Worth exploring

Files：apimart-gateway/main.py:_TASK_META · localTool/src/routes/tasks.ts

Problem：网关用内存字典 _TASK_META 存任务轮询状态（重启即丢，webhook 守护也随进程消失）；localTool 又有独立 tasks 表。两份任务记录各管各，seam 之间无契约对齐——前端轮询网关拿状态，任务清单却读 localTool 表，两者可能不一致。

Solution：明确"网关=运行时轮询状态（可丢），localTool=tasks 表=持久化任务档案（不丢）"的职责边界，写进 ADR；或让网关在终态时把结果写回 localTool tasks 表，统一真相源。

Benefits：Leverage——前端只需信 localTool 表的终态；Locality——任务真相单点。

④ /api/jianying/send 是 pass-through 空壳

Speculative

Files：localTool/src/routes/system.ts:160-193

Problem：该端点只 console.log + 返回 {status:'ok'}，无任何真实剪映集成。删除测试：删掉它，复杂度 vanish（没有调用方依赖它的真实行为，画布只收到假成功）。是典型 Shallow Module——接口（POST 端点）和"无实现"几乎同复杂。

Solution：要么补真实集成（写剪映草稿目录），要么在契约/文档明确标"未实现、返回假成功"，避免画布误以为发送成功。

Benefits：消除"假成功"误导；若保留则让它成为真实 deep module（小接口藏集成逻辑）。

⑤ upsert 非原子（DELETE+INSERT 两语句，无事务）

Worth exploring

Files：resources.ts:160-166 · tasks.ts:42-48 · database.ts

Problem：upsertResource/upsertTask 先 DELETE 再 INSERT，两条独立 run()。中途崩溃留空行；且每次 save 全删全插，无增量更新。Locality 差——原子性应内聚在 db 层，而非每个路由自己拼。

Solution：在 database.ts 提供 upsert(table, row)，内部用 sql.js 的事务（BEGIN/COMMIT）包裹，或改用 INSERT ... ON CONFLICT（sql.js 支持）。

Benefits：Locality——原子性集中；路由代码变浅（一行 upsert）；可测试性提升。

⑥ proxy 超时魔法数重复，未走配置 seam

Speculative

Files：system.ts:67 与 system.ts:126（FormData/JSON 两分支各写 15000）

Problem：15s 超时在两个分支硬编码重复，且未走 config。Seam 处的 adapter 配置（超时）散落 handler 内部。

Solution：抽 PROXY_TIMEOUT_MS 常量或读 config；两分支共用一个 proxyFetch(targetUrl, opts) 辅助函数。

Benefits：去重；Locality——超时策略单点。

Top 推荐

优先处理 ① localTool 持久化 seam 缺失。它是唯一会导致静默数据丢失的摩擦点（其他多为可维护性问题），且修复能立即提升 Locality（落盘内聚 db 层）。其次是 ② 端点硬编码——它正是 HANDOFF4 死循环类契约错配的根因模式，修掉能防复发。
