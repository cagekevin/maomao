## ① 运转图景

网关 FastAPI(:9004) 把 OpenAI 风格请求翻译为 Lovart 调用。生图/视频为异步：提交返回 `task_id` → 前端轮询 `GET /v1/tasks/{id}` → 网关 `_check_and_fire_task`(L878) 内部轮询 Lovart 并自动确认（`AUTO_CONFIRM=true` 默认）。

## ③ 关键数据流（网关轮询内部）

```
_check_and_fire_task(task_id, client)  (L878 调用)
  → Lovart poll loop:
      status==done → get_result(thread_id) (L506)
        pending_confirmation?
          AUTO_CONFIRM=true → client.confirm (L509/L523/L532) ✅ 自动确认
          AUTO_CONFIRM=false → raise _pending_confirmation_error (L512/L526/L535) ❌ 抛错
      status==abort → raise LovartError (L502/L521)
      else → asyncio.sleep(3) 继续 (L539)
```

## 对抗审计（门4）

**Q：前端 pending_confirmation 卡死真会发生吗？** A：默认 `AUTO_CONFIRM=true`(L34)，网关 L509/L824 自动确认，前端 L33058 不触发。仅当运维显式设 `AUTO_CONFIRM=false` 时，网关 L512 抛错 → 前端 markNeedsConfirm→return 永久 await_confirm。网关 L882 `confirm` 端点提供外部解救，但前端无对应 UI 触发。✅（B6 归因清晰）

**Q：completed vs done 词不一致会漏判吗？** A：**不会**。`_check_and_fire_task`(L783) 在 L799 `if status=="done"` → L804 先置 `running` 做 5s 防抖(L801-806) → 最终 L867 `_task_view(task_id,"completed",100,...)` 返回前端。前端 L33028 命中 `completed` ✅（待查项已关闭）

**Q：网关会永久挂起吗？** A：网关 `poll_count`(L811) 无硬上限，但每次 `get_task`(L878) 调用都重新进入，靠前端 15min deadline(L32997) 兜底退出。网关侧 `done_first_seen` 防抖已处理 done↔running 抖动。✅
