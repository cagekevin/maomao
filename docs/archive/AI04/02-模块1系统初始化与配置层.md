## ① 运转图景

启动链：`main.tsx` → `lazy(App.js)` → 顶层 `import` 映射（`App.js:L3` 巨长 import，把 config.js 导出重命名为单字母，如 `H=On`、`R=ge`，这是反编译残留，**禁止改**）→ 运行时 `var Hr = localEngineBase()`(L1732) / `var vv = LOCAL_ENGINE.base`(L42808) 初始化 base URL。

`StorageManager`(`Q`，混淆名，func-mapping 未收录但 ARCHITECTURE L1.4 称 `Q`) 在 L1418/L1446 装配 `Nr`(localStorageBackend) / `Pr`(localforageBackend) 两个后端，统一经 `jr`(L1345 cachedGet) 带缓存读取。

## ③ 关键数据流

- **去登录**：`Oa()`(L3543) 永远返回本地 token（PROJ_ORIGIN §4.5），`Ba()`(L3637) 恒 true → 所有 `Jh()`(L30712) 带本地 Bearer。
- **base URL 双源**：`Hr=localEngineBase()`(L1732, 随 USE_LOCAL_ENGINE 切换 18080/9004) vs `vv=LOCAL_ENGINE.base`(L42808, 固定 18080)。**这是交叉风险点3根因**（见 `06`）。
- **配置加载降级**：config.js 全 `UPPER_SNAKE`（合规），无运行时兜底缺失（PROJ_ORIGIN §8.4 现状合规）。

## ④ 存疑 Bug

| 项 | 状态 | 说明 |
|----|------|------|
| `USE_LOCAL_ENGINE=false` 时文件走 9004 无上传路由 | P2（交叉） | `Hr` 变 9004 但 `vv` 仍 18080；资源写库走 `vv`(18080) 安全，但 `Zr/ii` 上传走 `Hr`(9004) 失败。详见 `06` X3.3 |
| `Q`(StorageManager) 未在 func-mapping 收录 | 缺口 | 建议补录，本目录仅记 |

## 边界契约

- **入**：`config.js`(L1-L75) 全部导出；`chrome.storage.local`(L1367)；`localforage`(L1446)
- **出**：`auth_token`(localStorage)；`canvas-state-v1-{projectId}`(localforage, Cr L1294)；`StorageManager.getConfig/getObject`(经 `jr` L1345)
- **config 开关**：`USE_LOCAL_ENGINE`(config L36) → 决定 `Hr`(L1732)

## 对抗审计（门4）

**Q：Oa 真的是本地 token 不是真登录？** A：grep `Oa(` 调用点均走 `Jh`/`Ba`/`Zh`(L30724 `if(!Oa()) throw`)，无真实 OAuth 调用；符合 PROJ_ORIGIN §4.5「去登录」。✅

**Q：Hr 与 vv 不一致会破哪？** A：资源写库 `Sv/wv/Ev` 全用 `vv`(18080) 安全；仅上传 `Zr/ii` 用 `Hr`。`USE_LOCAL_ENGINE=false` 时 `Hr=9004` 上传失败但写库不崩。✅（归因见 `06`）
