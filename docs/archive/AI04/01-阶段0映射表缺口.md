## 2. 文档误用纠正（已坐实，原文档未改）

### 2.1 `Zr` 不是上传函数，是「下载远程URL落盘」
- `TASKS.md` 行34 / `ARCHITECTURE.md` L2.5 把 `Zr()` 当上传函数。
- **真相**：`Zr`(L1827) 形参 `e=fileUrl`，调 `POST /api/files/upload` 把**远程 URL** 抓到本地 `18080` 落盘。它是「远程→本地」下载器，对应 `uploadFile` 的远程分支。`ii`(L1888) 才是统一入口，`Xr`(L1802) 是 File 分支。
- 影响：`ARCHITECTURE.md` L2.5 表「写入入口 1/5」的 `Zr()` 表述需改为 `ii()→Zr()` / `ii()→Xr()`。

### 2.2 `Jn` = LogoIcon，不是生图主回调
- `TASKS.md` / `ARCHITECTURE.md` 多处以 `Jn` 指代「生图主回调 / AI 生成」。
- **真相**：`Jn`(App.js L89) = `function Jn({size}) { ... 品牌 SVG ... }`（func-mapping L65 `LogoIcon` ✅）。真实 AI 生成派发在 **L32987 区域**的一个大 `useCallback` 内（依赖 `R`/`Jr` 等），不是独立 `function Jn`。
- 影响：`ARCHITECTURE.md` L1.3 `Jn ~L32731 生图主回调` 整行错；`TASKS.md` T2.3 `锚 Jn` 应改为「锚 L32987 生图派发 useCallback」。

### 2.3 `we` 不是模块级函数，是组件内闭包
- `func-mapping.txt` T0.1 把 `we` 当「rescanResources 主函数（模块级）」。
- **真相**：`we`(L43015) 是 `Y.useCallback` 闭包（资源面板组件内），内部调 `Ev()`(L42883)。模块级真身是 `Ev`。
- 影响：审计锚点用 `Ev`(L42883)，不用 `we`。

**Q：Zr 真的是下载器不是上传器？那「发送到资源」落盘走谁？**

**Q：Jn 真是 LogoIcon，那生图派发函数叫什么？**
