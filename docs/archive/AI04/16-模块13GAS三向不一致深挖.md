## 二、三向不一致的**根因**（grep 坐实）

### 2.1 推送白名单 vs 拉取全量（键集合不对称）
- **推送** `ei`(L43954)：固定 9 键白名单
  `app_settings / api_configs / users / membership / projects / presetPrompts / customNodeTemplates / modelSchedules / cloud_storage_config`
- **拉取** `ti`(L43992–L43994)：遍历云端 `Object.keys(t)` **无限制**，逐个 `Q.setObject(e,n)`。

> 不对称后果：云端若含非白名单键（如旧版 `transitResources`、`canvas-state`、`localforage` 迁移残留等），拉取会**无差别写回本地**，但下次推送永不带上这些键 → 本地↔云端单向泄漏，三向（本地 kvStore / 云端 / localforage）不闭合。

### 2.2 `modelSchedules` 走特殊通道
- 推送 L43955：`t === 'modelSchedules' ? la() : await Q.getObject(t)` —— 推时用内存 `la()` 而非 kvStore。
- 拉取 L43994：`e === 'modelSchedules' ? await Sa(n) : await Q.setObject(e,n)` —— 拉时用 `Sa`（独立状态写入）而非 kvStore。
> `modelSchedules` 在推送/拉取均与 kvStore 旁路，若 `la()`/`Sa()` 与 `Q` 状态不一致会偏离。

### 2.3 拉取后强制 reload
- L43997 `window.location.reload()`（延迟 1s）。拉取成功后整页刷新以应用 kvStore 变更——但若上面 2.1 写回了非预期键，reload 后可能污染运行态。

### 2.4 推送缺 `transitResources` / 画布状态
- 导出 `Ri`(模块7 L44511) 含 `lastOpenedProject` + 画布状态，但 GAS 推送白名单(L43954) **不含** `lastOpenedProject` 与画布状态。即：配置可云同步，但「当前打开项目/画布」不同步 → 换设备拉取后需手动重开项目。

## 三、结论
GAS 三向不一致不是单一 bug，而是**推送白名单窄 + 拉取无校验 + modelSchedules 旁路 + 画布态不参与**四者叠加的设计缺口。最危险项是 2.1（拉取无键白名单校验）。
