## 一、GAS 云端同步真实形态（var-mapping L9–L12 误植）

### 1.1 真实实现：`CloudSyncEngine`（非 `ei`/`ti` 符号）

App.js **L43897** `CloudSyncEngine = { config:{gasUrl: GAS_CLOUD_SYNC_URL}, isSyncing:false, async callGateway(action,data){...} }` —— 这是 GAS 云端同步引擎。`isSyncing` 是其**实例属性**（L43901/L43905/L43923），**不是** `Qr`/`$r` 符号。

- 推送：`CloudSyncEngine.push(e, ...)` @ **L43962**（被 L43950 `ei` 调用）
- 拉取：`CloudSyncEngine.pull(...)` @ **L43977**（被 L43974 `ti` 调用）

### 1.2 `ei` 的三个含义（var-mapping 只收其一且错）

| 位置 | 含义 | 证据 |
|------|------|------|
| App.js **L1853** `var ei = new Map()` | 模块级**缓存 Map**（缩略图/资源缓存，被 `ri` L1862/L1877/L1885 读写）| 与 `ti`(L1854) 成对，`ri`=缩略图缓存 |
| App.js **L36784** `ei = Y.useCallback(()=>{...})` | 组件级**「打组」**函数（group nodes，`group-${Date.now()}`，Ctrl+G 触发 L36962）| 过滤 selected 节点建 group |
| App.js **L43950** `ei = async () => { $r(true); ... CloudSyncEngine.push }` | 组件级**真正 pushToCloud**（收集 app_settings/api_configs/... 同步）| 调 `CloudSyncEngine.push`(L43962) |

→ var-mapping L9 `ei = pushToCloud` **只命中 L43950 一种**，却把它当作模块级语义，忽略了 L1853(缓存Map)/L36784(打组)。**引用 `ei` 必须带行号，且模块级 `ei` 实为缓存 Map 而非推送。**

### 1.3 `ti` 的三个含义

| 位置 | 含义 |
|------|------|
| App.js **L1854** `var ti = new Map()` | 模块级缓存 Map（与 `ei` 成对）|
| App.js **L36822** `ti = Y.useCallback(async () => { 扫描画布缓存/清理 })` | 组件级**清缓存**函数（调 `Kr()`=localTool 连接检测）|
| App.js **L43974** `ti = async () => { ... CloudSyncEngine.pull }` | 组件级**真正 pullFromCloud** |

→ var-mapping L10 `ti = pullFromCloud` **只命中 L43974**，忽略 L1854(缓存Map)/L36822(清缓存)。

### 1.4 `Qr`/`$r` 的三个含义

| 符号 | 位置 | 含义 |
|------|------|------|
| `Qr` | App.js **L1847** `function Qr(e)` | 模块级**文件类型前缀**（`img`/`vid`/`aud`/`file`，L1809 用 `Qr(n.type)`）|
| `Qr` | App.js L36626/L43949 | 组件级 **useState(bool)** 或 useCallback（UI 同步中标志）|
| `$r` | App.js **L1850** `function $r()` | 模块级 **随机 uid**（`Math.random().toString(36).slice(2,8)`，L1809 用 `$r()`）|
| `$r` | App.js L36724/L43949/L43951 | 组件级 **setSyncing UI setter**（L43951 `$r(true)` 置同步中）|

→ var-mapping L11–L12 `Qr`=isSyncing / `$r`=setSyncing **完全错植**：`isSyncing` 实为 `CloudSyncEngine` 实例属性（L43901），`Qr`/`$r` 模块级是文件名/uid 助手，`$r` 的"setSyncing"语义仅存在于**组件局部 useState**，非模块级符号。

### 1.5 判定

❌ **var-mapping L9–L12 大面积误植**。GAS 云端同步的真实入口是 `CloudSyncEngine`(L43897) + 组件级 `ei`(L43950 push)/`ti`(L43974 pull)，`isSyncing` 是其属性。建议映射表：
- 删除/修正 `ei`/`ti`/`Qr`/`$r` 的 GAS 收录；
- 改标 `ei`(L1853)/`ti`(L1854)=缓存 Map，`Qr`(L1847)=文件类型前缀，`$r`(L1850)=随机 uid；
- 新增 `CloudSyncEngine`=GAS 云端同步引擎（L43897），`push`(L43962)/`pull`(L43977)；
- 注明 `ei`/`ti` 组件级重定义（pushToCloud/pullFromCloud/打组/清缓存）须带行号。

## 二、跨进程 `resourceAdded`（background.ts 实测）

**08 结论**：`resourceAdded` 非 window CustomEvent，源 = `background.ts` `chrome.runtime.sendMessage({action:'resourceAdded'})`，前端以 `e.action==='resourceAdded'` 接收（App.js L43527）。

**实测 background.ts L101–L102**：
```ts
chrome.runtime.sendMessage({ action: 'resourceAdded', resource: newResource });
```
→ 确认 08 结论。**注意**：`background.ts` 是 Chrome 扩展 service worker，本地模式（localTool 直连）下此路径**不触发**；资源新增在本地模式走 `ii`(L1888)/`uploadFile`(L19098) → `/api/files/upload` 直接入库，`resourceAdded` 跨进程通知仅扩展态生效。建议在 08 补注"本地模式不依赖该跨进程事件"。

## 三、端口一致性（jianying vs upload）

**13 遗留**：上传走 `Bc`、剪映走 `Wn`，是否同端口？

**实测**：
- App.js **L19049** `var Bc = LOCAL_ENGINE.port` → config L14 `port: 18080`。
- App.js **L7** `Wn = JIANYING_PORT` → config L21 `JIANYING_PORT = 18080`（注释「已禁用，保留定义」）。
- 剪映发送实际 fetch `http://127.0.0.1:${Wn}/api/jianying/send`（L14/L60）。

**判定**：✅ 二者同为 18080（LOCAL_ENGINE.port = JIANYING_PORT = 18080），端口一致；剪映发送当前被 config 注释"已禁用"，非运行路径。无端口漂移。
