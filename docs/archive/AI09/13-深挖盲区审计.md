## 一、剪映发送链路：`Kn` ≠ sendToJianying（映射表误植）

**TASKS / func-mapping 现状**：`func-mapping.txt` L134 记 `Kn = sendToJianying`（发送到剪映本地 API：`http://127.0.0.1:${Wn}/api/jianying/send`）。

**实测**：
- App.js **L44** `function Kn(e, t='mp4')` 是**文件名清洗器**：取 URL 末段文件名，无扩展名则补 `clip_${Date.now()}.${t}`。被 L119/L29285 以 `fileName: Kn(e.url,'png')` 调用。
- 发往剪映的真实逻辑在 **`qn`**（App.js **L53** `async function qn(e)`，内部 `fetch('http://127.0.0.1:${Wn}/api/jianying/send', POST, application/json)` @ L60），以及组件内 `Qn` useCallback（App.js **L33383**）。
- App.js 全量 grep 字符串 `sendToJianying` = **0 匹配**——该名字根本不是 App.js 中的符号。

**判定**：❌ `func-mapping.txt` L134 `Kn = sendToJianying` **错误**，须修正：
- `Kn`(L44) = 文件名清洗器（建议名 `sanitizeClipFileName`）；
- 剪映发送逻辑 = `qn`(L53) / `Qn`(L33383)，建议映射表收录 `qn`=sendToJianying（或项目内实际导出名）。

## 二、`Qn` 同名碰撞（与 `Jn`/`Lr`/`Sv` 同型）

**现状**：`func-mapping.txt` L66 `Qn = TransitPanel`（资源转存面板组件，定义于 App.js L169）。

**实测**：同一混淆名 `Qn` 在组件作用域内被**重定义**为 SD2 视频生成回调（App.js **L33383** `Qn = Y.useCallback(async (e,t,n,r,i)=>{...})`，引用 `m`=API Key、`W`=setNodes）。

**判定**：⚠️ `Qn` 存在模块级（TransitPanel）+ 组件级（SD2 回调）双义。引用 `Qn` 必须带行号（同 `Jn`/`Lr` 处理方式）。映射表已收模块级，组件级 `Qn` 须带行号引用。

## 三、pending_confirmation 确认流（TASKS「其他限制」坐实）

**TASKS 现状**：「单任务 `pending_confirmation` 卡轮询（AUTO_CONFIRM=false 时）」列为无害限制。

**实测**（AI 生成轮询核心）：
- 检测：`resolveNeedsConfirm(taskInfo)` 返回 `taskInfo?.error?.code === 'pending_confirmation' ? taskInfo.id : null`（App.js **L32958**）。
- 退出轮询置等待：注释 `[架构5步法] pending_confirmation：退出轮询，节点置等待确认`（L33058/L33545/L34217）。
- 确认适配器：`makeConfirmAdapter(gatewayBase, authToken)` → `zc(\`${gatewayBase}/v1/tasks/${taskId}/confirm\`, {method:'POST', headers:{Authorization:'Bearer '+authToken}, localPort: H.status.isConnected?H.status.port:undefined})`（App.js **L32960–L32967**，确认 URL = `${R}/v1/tasks/${taskId}/confirm`）。

**判定**：✅ TASKS 描述成立。确认走 `zc`(fetch 原语, L19001) POST `${R}/v1/tasks/{id}/confirm`；`AUTO_CONFIRM=false` 时停等人工确认，不无限轮询。06 模块3 的 confirm 锚点(L32963)坐实。

## 四、双上传路径（落盘 18080 的两条链路）

**实测**：
- 路径 A（直接 FormData）：`ii`(App.js **L1888**) 内 `new FormData()`(L1810/L1830) → `fetch(\`${Hr}/api/files/upload\`)`(L1812/L1832)。即模块2/3 拖入与资源落盘走 `ii`。
- 路径 B（StorageManager 封装）：`uploadFile`(App.js **L19098** `uploadFile: Y.useCallback(async (t,n,r='')=>{...})`，H=StorageManager) → `fetch(\`http://127.0.0.1:${Bc}/api/files/upload\`)`(L19102)。被 AI 生成结果回填调用（L34699/L35279/L35372/L44377 `H.uploadFile(e,t,'tasks')`）。

**判定**：✅ 存在两条上传链路，均打 localTool `/api/files/upload`（index.ts L135）。`ii` 偏资源/拖入，`H.uploadFile`(StorageManager) 偏任务结果回写。模块3 的 `uploadFile` L19098 锚点坐实；建议文档注明二者并存。

## 五、`handleUploadFormData` 是 localTool 服务端符号（TASKS 上下文缺失）

**TASKS 现状**：P1-4「`handleUploadFormData` fileUrl 跨域失败 — 实际不是问题（Node fetch 不受 CORS 限制）」。

**实测**：App.js 全量 grep `handleUploadFormData` = **0 匹配**；该符号实存在于 **localTool 服务端**：`localTool/src/routes/files.ts` L25 `async function handleUploadFormData(req,res)`（被 L19 分发调用），以及 `system.ts` L30 `handleProxyFormData`（X-Proxy 头代理转发）。

**判定**：⚠️ TASKS 把服务端符号当前端上下文呈现，无文件标注。结论（Node fetch 不受 CORS）本身正确，但易误导前端审计者去 App.js 找符号。建议 TASKS 标注「localTool 服务端 `files.ts` L25」。

## 六、`zc` = 统一 fetch 原语（被低估的核心）

**实测**：App.js **L19001** `async function zc(e, t)` —— 处理 `body instanceof FormData`/`Blob`（L19002–L19003）、`localPort` 路由（L19004 拼 `http://127.0.0.1:${t.localPort}/api/files/read`）、以及标准 fetch。是 `tr`/`ar` 之下的**真正 HTTP 执行层**，被 confirm(L32963)、资源读取、代理转发等广泛调用。

**判定**：🟡 映射表未收录 `zc`。它是比 `tr`/`ar` 更底层的 fetch 原语，建议补入 func-mapping（如 `zc`=proxyAwareFetch / localPortFetch）。
