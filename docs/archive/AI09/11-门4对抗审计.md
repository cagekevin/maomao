## 质询 1：画布持久化机制——`canvas-state-change` 虚构说是否成立？

**原结论（03/07/08）**：ARCHITECTURE X2 表的 `canvas-state-change` CustomEvent 在 src 全目录 0 匹配，画布持久化非事件总线驱动。

**反向质询**：那 ReactFlow 的节点变更到底怎么落盘？`Pr`(localforageBackend) 被谁直接调用来存画布？

**grep 实证**：
- `canvas-state-change` 全目录 **0 匹配** ✅（虚构坐实）
- 真实链路：ReactFlow `onNodesChange: fe`(App.js **L37123**) → 受控 state(W/G) → 变更检测后 `r.nodes = e`(App.js **L31722**) → `Q.saveCanvasStateWithVersion(en.current, r, t)`(App.js **L31728**)
- `Q.saveCanvasStateWithVersion` 是 StorageManager 方法(App.js L1650)，`Q`=StorageManager(var-mapping L20)，底层 `Pr`(localforageBackend) 由其编排，并写 `canvas-state-v1-{projectId}`(前缀 Z.CANVAS_STATE_PREFIX L1283，L1599 匹配)
- `mutiwindow-task-completed` 监听处(App.js L31428) handler 只更新节点 data，**不**触发持久化（持久化另有 onChange 链）

**判定**：✅ 原结论成立并更精确——持久化经 `Q.saveCanvasStateWithVersion`(StorageManager, L31728)，非 CustomEvent、非直接 `Pr` 写。07 §3.3 已据本次实测修正。

## 质询 3：`Sv`(大写) 与 `sv`(小写) 是否真为不同物？门3 为何 Sv 匹配到 L5367？

**原结论（10）**：`Sv`(大写)=资源入库 async function(L42838)；`sv`(小写)=nodeCallbackFieldSet Set(L42002)，须大小写区分防混。

**grep 实证**：
- `async function Sv(e)` **仅 L42838 一处定义** ✅
- L5367/L8775/L43107/L44168/L43552 的 `Sv({...resource, url, folder:'migrated'...})` 均为**调用点**（migrated 迁移路径 + 统一入库）
- `sv`(小写) 仅 L42002 一处：`new Set('id.label.type...'.split('.'))` = 节点回调字段 Set ✅
- 门3 脚本 FUNC_CALL_RE 命中 L5367 是"调用点存在"而非"定义行"，属软匹配噪声，不影响"符号存在"判定

**判定**：✅ `Sv`(大写)/`sv`(小写) 确为不同物。10 的大小写区分建议成立。L5367 是调用点非定义，无双义。

**判定**：✅ localTool 已具备删磁盘能力（参数 `deleteFiles`），`wv`(L42857) 不传即只删 DB。09 结论"修复成本低（wv 传 deleteFiles:true）"成立。
