## 一、运转图景（Module 3：AI 生成与网关层）

用户在节点触发生成 → 前端 `Jn`(组件内生图主回调) 向网关 `:9004` 发请求 →
- **同步分支**：直接拿到图片 URL（或 SSE 流式逐块返回）。
- **异步分支**：响应含 `task_id` → 自建轮询 `GET /v1/tasks/{id}`（≥3s/退避/15min deadline）→ 拿到 `results[0].url` →
  经统一同步 effect `uploadFile` 落盘 `uploads/tasks/` → 派发 `mutiwindow-task-completed` → rescan 刷资源面板。

> ⚠️ **关键澄清（修正 TASKS T0.1）**：TASKS 假设「真实 AI 生成派发函数（非 Jn）」是独立顶层函数。
> 实际 grep 证实：**派发逻辑嵌在 `Jn`@L32490 主回调内部**（非独立函数）。
> `Jn`@L32490 是组件内 `Y.useCallback`（遮蔽模块级 `Jn`=LogoIcon@L89）。
> 异步分流在 `Jn` 内 `if (N)` 分支：L32986 `task_id` 检测 → L32990 轮询循环。
> 即「派发函数」= `Jn` 内部 N 分支，无独立顶层名。引用必须带 L32490 + L32986。

## 三、关键数据流（AI 生成端到端）

```
Jn()(App.js L32490, 节点 onGenerate 触发)
  → 校验模型权益 Xi()/Ai()(L32493/L32495)
  → fetch(网关)
      同步：SSE 流式 → 解析 results[0].url (L32916-32945)
      异步：if(N) → task_id 检测(L32988)
            → while: zc(GET ${R}/v1/tasks/${taskId}) (L33005/L33007)
                 ≥3s 间隔 / 退避 / 15min deadline (L32997-32999)
                 pending_confirmation → requestConfirm() (L32957-32979)
            → 取 results[0].url (数组[0])
  → 更新节点 data.imageUrl/videoUrl
  → 入 globalTasks (updateGlobalTasks, L15157+)
  → 派发 mutiwindow-task-completed (L38481, 带 resultUrl)

统一同步 effect (App.js L44352, 依赖 [globalTasks, isConnected, port])
  → 锁 Se.current
  → 遍历 globalTasks(status=completed):
       未本地化(非 /files/ 开头) → H.uploadFile(url, name, 'tasks') (L44377)
       有变更 → await Di.current() 即 we()(rescan@L43015) 刷资源面板
  → finally 释放锁
```

## 四、存疑 Bug（对照 TASKS + 红线 §3.2，门4 质询点）

| # | 存疑点 | 证据 | 门4 质询 |
|---|--------|------|---------|
| 轮询-1 | 轮询 ≥3s/退避/15min（L32997-32999）符合红线异步契约 | L32997 `9e5`=15min | "deadline 后与网关内存任务库(TTL 86400s)是否冲突？超时任务网关是否已清 → 轮询永远 failed？" |
| 轮询-2 | `pending_confirmation` 卡死：AUTO_CONFIRM=false 时需手动 confirm（L32957-32979） | TASKS 已知限制 | "confirm 走 `zc(${gatewayBase}/v1/tasks/{id}/confirm)`；前端 `markNeedsConfirm` 置 await_confirm 后，谁触发 confirm？grep confirmTaskId 消费点" |
| 红线-1 | **CDN 直链违规风险**：若 `results[0].url` 是 CDN 地址且统一同步 effect 未触发 `H.uploadFile` → 节点直接显示 CDN 链接，刷新即丢 | L44377 仅未本地化才落盘 | "url 已以 /files/ 开头（127.0.0.1）才跳过；CDN https 必进 uploadFile ✅。但若 globalTasks 未含该任务 → 跳过同步 → 破图。grep 写入 globalTasks 的所有路径" |
| 轮询-3 | `R`(网关base) 局部变量来源：生图作用域 `R` 值由谁赋值？是否等于 `DEFAULT_ENDPOINT`(9004) | L33005 | "grep 生图组件 `R=` 赋值，确认不等于 localTool 18080（混用会 404）" |
| 落盘-1 | 三个落盘入口并存：`ii`@L1888 / `Xr`@L1802 / `H.uploadFile`@L19098，调用方混乱 | 见 T2.2 + 本文 | "统一同步用 `H.uploadFile`(L44377)，resourceAdded 链用 `ii`(L1888)→`Xr`(L1802)；是否应统一到 `ii`？是否存在某路径漏落盘？" |

### 门4 对抗审计结论（自审）
- **轮询-3 需上游确认**：`R` 在生图作用域是局部变量，grep 其赋值点（L32490 闭包外 `R` 来自组件 props/context）确认指向 9004。本文无法在单遍内坐实其值，标注待查。
- **落盘-1 是真实架构债**：三个入口语义重叠，`H.uploadFile`@L19098 挂在 localTool 连接 context，与 `ii`@L1888 功能近似但独立实现，存在重复与潜在不一致。建议统一收口到 `ii`。
- **红线-1 当前安全**：统一同步 effect 对 https/CDN url 必走 `H.uploadFile`，不破图；风险仅在 globalTasks 漏写。

## 五、边界契约（缝合 T2.5.1 用）

### 5.1 发出的 HTTP 路由（网关 :9004）
| 路由 | 方法 | 调用方 | 说明 |
|------|------|--------|------|
| `/v1/images/generations` 等 | POST | `Jn`@L32490 内 fetch | 生图/视频/音频（见 ARCHITECTURE L3.2） |
| `/v1/tasks/{id}` | GET | `zc`@L33007（轮询） | 异步任务状态 |
| `/v1/tasks/{id}/confirm` | POST | `zc`@L32963 | 手动确认（AUTO_CONFIRM=false） |
| `/v1/chat/completions` | POST | SSE（流式） | 聊天 |

### 5.2 跨层：localTool 落盘（统一同步触发）
| 路由 | 方法 | 调用方 |
|------|------|--------|
| `/api/files/upload` | POST | `H.uploadFile`@L19098（`zc` 代理走 `/api/proxy`@L19016） |

### 5.3 发出的 window CustomEvent
| 事件 | 触发(行号) | 消费 |
|------|-----------|------|
| `mutiwindow-task-completed` | L38481（带 resultUrl） | 资源面板 rescan（L31428 监听）→ 接 T2.2 模块 |
| `mutiwindow-rerun-task` | L44343 | 重新执行任务 |

### 5.4 依赖 config
| 配置 | 影响 |
|------|------|
| `DEFAULT_ENDPOINT`/`ENDPOINTS`(config.js) | 网关 9004 基址（`R` 生图 base 应源自此） |
| `USE_LOCAL_ENGINE` | true→落盘走 18080（`H.uploadFile` 依赖 `n.status.isConnected`） |

## 六、结论
1. TASKS T0.1 的「非 Jn 真实派发函数」假设**不成立**——派发嵌在 `Jn`@L32490 内 N 分支（L32986+），无独立顶层名。建议上游修正 T0.1 表述。
2. `R` 同名遮蔽第四例坐实（4 种语义），强烈建议根映射表补 `R(globalTasks)`@L31304 / `R(gatewayBase)`@L33005 标注。
3. 统一同步 effect 真实行号 **L44352**（非 ARCHITECTURE 标 L44246，漂移 +106）。
