## 0. 本文缝合的 4 条端到端流

| 流 | 起点 | 终点 | 关键缝合点 |
|---|---|---|---|
| **流A** AI生成→统一入库 | 节点触发生图 `Jn`(L32490) | 资源面板显示 | `mutiwindow-task-completed` dispatch→listen→`Ev`+`Di.current` |
| **流B** 外部素材→本地化入库 | 网页/扩展 `resourceAdded` 消息 | `migrated/` 落盘+面板 | `chrome.runtime.onMessage`→`Zr`(uploadFromUrl)→入库 |
| **流C** 画布拖放→本地落盘 | 画布 `onDrop` `Lr`(L36293) | `canvas/drop` 目录 | `ii`(uploadFile, L1888) 落盘（**不自动进面板**） |
| **流D** Rescan→面板刷新 | `Di.current()`=`we`(L43015) | 面板数据 `D` 更新 | `Ev()`(rescan, L42883)→`xv`(拉列表)→`D` |

> 模块1-4 已分别审计各段；本文验证**段间接缝是否真实连通**。

## 1. 流A：AI生成 → 统一入库（最完整的一条）

### 1.1 链路（源码坐实）

```
节点 onGenerate=sr / onGenerateVideo=ur ...  (Z 注入, App.js:L35572-L35576)

        window.dispatchEvent(new CustomEvent(`mutiwindow-task-completed`, {...}))
        位置: App.js:L38481 (主路径) / L43640 / L43676 / L43697 / L44406
   │
   ▼  [CustomEvent 跨函数边界]
listen (App.js:L31428, window.addEventListener)
   │
   ▼  i === `completed` && n 时 (App.js:L31426):

         ├─ Ev() = rescan 派发 (App.js:L42883 → /api/resources/rescan)
```

### 1.2 结论
流A **闭合且自洽**：生成完成事件同时触发 `Ev`(后端 rescan) 与 `Di.current`(前端拉列表刷新)，面板在生成完成后自动出现新资源。这是四条流中唯一**全自动**的。

## 2. 流B：外部素材 → 本地化入库

### 2.1 链路（源码坐实）

```
扩展/网页侧: chrome.runtime.sendMessage({action:'resourceAdded', resource})
   │
   ▼
listen: chrome.runtime.onMessage.addListener(r) (App.js:L43564)
   │
   ▼  r 内: if (e.action === `resourceAdded` && e.resource) (App.js:L43527)
        let r = t.source === `local-tool`;
        if (!r) {  // 非 local-tool 来源（网页直链/data:/http）
           let localized = await Zr(i.url, {subfolder:`migrated`});  // uploadFromUrl L1827
           if (!localized?.url) return n({success:false,error:`download_failed`}), false;
           i.url = localized.url.startsWith(`http`) ? localized.url : `${Hr}${localized.url}`;

   │
   ▼  入库 → Te(`transit`); j(`materials`) (App.js:L43555)
```

### 2.2 结论
流B 通过**扩展消息通道**（非 React 函数直连）入库，`Zr`(uploadFromUrl, L1827) 把外部地址下载到 `migrated/` 子目录，符合模块2「本地化铁律」。注意：此处 `j('materials')` 直接切到素材面板，但**是否触发列表重拉取决于 `D` 是否被赋值**——`Zr` 落盘后同样需要 rescan 才能被面板列表 `xv` 拉到（除非 `resourceAdded` 处理内另有 `D` 写入）。当前 L43555 只切 tab，未显式 `D(...)`，故实际显示依赖后续 rescan。

## 3. 流C：画布拖放 → 本地落盘（不自动进面板）

### 3.1 链路（源码坐实）

```
用户拖文件到画布 → onDrop: Lr (App.js:L36293)
   ├─ 本地文件分支 (App.js:L36337):
   │     let t = await ii(e, {subfolder:'canvas/drop', preferThumbnail:l, thumbMaxDim:480, thumbQuality:75});
   │     // ii = uploadFile (App.js:L1888) → /api/files/upload 落盘
   │     if (t.url && /^https?:\/\//i.test(t.url)) Z(`imageNode`/`audioPlayerNode`, {imageUrl:t.url})  // 生成节点
   │     else fallback base64 (L36384)
   ▼
落盘成功 → 节点显示 imageUrl（内存态）
```

### 3.2 关键发现（与流D 对比）

- `ii`(L1888) 只是 `fetch(/api/files/upload)`，**不 dispatch 任何完成事件、不调 `Di.current()`**。
- 全局搜索 `Di.current()` 调用点仅两处：`App.js:L31426`(生成完成) 与 `App.js:L44123`(清空资源)。**画布 `onDrop` 落盘后无 `Di.current()`**。
- 因此：拖入画布的文件虽已落盘 `canvas/drop/`，但**资源面板不会自动刷新显示**——除非：
  1. 用户手动触发 rescan（模块2 `we`/调度），或
  2. 该节点后续经 AI 生成完成事件(L31426) 顺带 rescan。

### 3.3 结论 → 这是 P1 级缝合缺口
流C 与流D **未自动连通**：画布落盘 ≠ 面板可见。这是模块4 P1-画布-04（节点内 onDrop 不落盘）的**上游兄弟问题**——即使落盘了，面板也看不到。建议：在 `Lr`(L36293) 落盘成功后追加 `Di.current?.()`（或 `rescanThrottledSync`），使画布导入即时进面板（属 V1 优化，需评估红线）。

## 4. 流D：Rescan → 面板刷新（所有流的汇聚点）

### 4.1 链路（源码坐实）

```
Di.current = we  (App.js:L44350, 赋值 rescan 调度)
   │
   ▼  we = Y.useCallback(async () => {  (App.js:L43015)
        if (n.status.port) {
           await Ev();                          // rescan 派发 (App.js:L42883)
           D((await xv({page:1,pageSize:300,sortBy:'timestamp',sortDir:'DESC'})).items);
           // xv = 拉取资源列表 (模块2 已坐实) → D = 面板状态
        }
   })
   │
   ▼  另有 rescanThrottledSync (App.js:L43028) 节流 3s 防抖
```

### 4.2 结论
`we`(L43015) 是**面板数据唯一刷新入口**：先 `Ev()` 让后端重扫磁盘，再 `xv` 拉列表写 `D`。谁调用 `we`/`Di.current`，谁就能让面板看到新文件。当前调用方：流A(生成完成)、清空资源、节流同步。流C(画布落盘) **缺这一调用** → 印证流C 缺口。

| 流 | 触发源 | 落盘 | 触发 rescan | 面板可见 | 是否全自动 |
|---|---|---|---|---|---|

## 6. 存疑 Bug / 风险点（交叉流层面）

### P1-交叉-01：画布落盘与面板断裂（流C→D 缺口）
`Lr`(onDrop, L36293) 落盘 `canvas/drop` 后不调 `Di.current()`，资源面板看不到拖入的文件。与模块4 P1-画布-04 互为表里。**建议**：在 `Lr` 落盘成功后调用 `rescanThrottledSync()`（L43028）闭合链路。

### P2-交叉-02：流B 入库后未显式刷新列表
`resourceAdded` 处理(L43527) 末尾 `j('materials')`(L43555) 只切 tab，未 `D(...)` 写面板状态；实际显示依赖后续 rescan。若 rescan 未触发，用户切到素材面板可能看不到刚入库的项（直到手动刷新）。**建议**：入库后调 `rescanThrottledSync()`。

### P2-交叉-03：`mutiwindow-task-completed` 多 dispatch 点语义重叠
同一完成事件在 L38481/L43640/L43676/L43697/L44406 共 5 处 dispatch，listen 仅 L31428 一处。多 dispatch 点若携带 `nodeId` 不一致，可能导致 rescan 重复或漏触发。当前 listen 内 `i==='completed' && n` 统一处理，重复 dispatch 会被 `rescanThrottledSync`(L43028) 的 3s 节流吸收，风险可控（非 Bug，体验冗余）。

### P3-交叉-04：rescan 节流窗口 3s 可能掩盖并发落盘
`rescanThrottledSync`(L43028) 3s 内只跑一次。若画布连续拖入多个文件（流C），且未来补上 rescan 调用，3s 内的多次落盘只触发一次 rescan——但 rescan 是「全量重扫磁盘」，故最终一致，仅中间态延迟（非 Bug）。

## 7. 审计结论

1. **流A 全自动闭合**：生成完成事件 → rescan → 面板刷新，是设计最完整的一条。
2. **流B/C 与流D 断裂**：外部素材(流B)与画布拖放(流C)落盘后**均不主动触发 rescan**，面板不可见或依赖后续 rescan。这是当前 V1 的**核心缝合缺口**（P1-交叉-01 / P2-交叉-02）。
3. **流D 是单一汇聚点**：`we`(L43015) = `Di.current` 是面板刷新唯一入口，谁调用谁受益；当前调用方过少。
4. **修复方向统一**：在流B(L43555) 与流C(L36363 落盘成功后) 追加 `rescanThrottledSync()` 调用，即可闭合全部三条缺口，且复用既有节流防抖，改动局部、风险低。
