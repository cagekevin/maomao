## 1. 生图派发函数（T0.1 最关键纠正）

### 1.1 `Jn` 实为 `LogoIcon`，不是生图回调
`App.js` L89：
```js
function Jn({
  ...
  // 返回品牌 SVG（viewBox 0 0 1389 1024）
})
```
func-mapping.txt 第 65 行已正确记为 `Jn = LogoIcon`。**旧 ARCHITECTURE/TASKS 把「生图主回调」误标为 `Jn` 是错误的**。

### 1.2 真正的生图派发函数是 `Jn`(L32490) 这个 useCallback
`App.js` L32490：
```js
let Jn = Y.useCallback(async (e, r, o = `1024x1024`, s, c = `auto`, d = 1) => {
  // 取模型 → 校验 API Key → 设 loading → 调网关生成
  // 内部走 N 分支：POST /v1/images/generations → 拿 task_id → 轮询 → 落盘
})
```
- 证据：外层是 `Y.useCallback`，接收 `(nodeId, prompt, size, ...)`，设节点 `loading:true`，符合「节点生图回调」语义。
- **注意**：这又是同名 `Jn`——L89 是组件、L32490 是生图回调。引用须带行号。
- 轮询/落盘细节待模块3 审计时展开（FUNCTION_MAP §2.1 的 7 陷阱仍适用，但锚点须改用 L32490 而非「Jn 生图」泛称）。

## 2. 上传/下载函数歧义（T0.2 最关键纠正）

### 2.1 `Xr` 双重身份
| 行号 | 身份 | 证据 |
|------|------|------|
| L1802 | `uploadToLocalTool`（`[uploadHelper]`） | `fetch(${Hr}/api/files/upload)` + `subfolder` + 返回 `{url,thumbnailUrl,path}` |
| L43881 | `openInTab` | `Xr = async e => {...}` 活动标签打开 URL（func-mapping L173 录的是这处） |

`ii`(L1888) 内部对 **非 URL** 调用 `Xr(e,{subfolder})` → 走的是 **L1802 的上传版**。

### 2.2 `Zr` 双重身份
| 行号 | 身份 | 证据 |
|------|------|------|
| L1827 | `uploadFromUrl` | `fetch(${Hr}/api/files/upload)` + `fileUrl` 参数（按 URL 落盘） |
| L43893 | `logout`/`重置配置` | 设置面板 `Zr = async () => {...}`（func-mapping L48 录的是这处） |

> **旧 TASKS 行34 写「`Zr()` 下载」是错误的**——`Zr(L1827)` 是上传、`Zr(L43893)` 是重置配置，都不是下载。

### 2.3 资源面板 `R` / `B` 是局部名，非模块级
- `R` 在资源/图片导入多处以 `let R = Y.useCallback(...)` 局部声明（如 L29160 调用的 `R(e)` 是文件上传回调），**不是单一模块级函数**。func-mapping 跳过 `R`(局部) 的判断正确。
- `B`(L29188) 是图片节点 `onDrop` 回调：`let B = Y.useCallback(async e => {...})`，处理拖入文件/URL。URL 拖入只 `O([{url,source:'drop'}])` 存内存，**不落盘不入库**（印证 TASKS P1）。
