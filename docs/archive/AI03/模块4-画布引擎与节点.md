## 0. 与旧文档的纠正（T0 阻塞项延续）

旧文档（`docs/AI04`、`AI09`、`AI11` 等）对「spawnable 集合」存在**严重误读**，在此统一纠正：

| 旧文档说法 | 源码真相 | 证据 |
|---|---|---|
| `spawnable`(L4141) 是「可拖放生成的节点白名单」，仅 13 种，缺 `textNode/audioNode/textConcatNode` 故不可生成 | `spawnable` 实际是**「工作流资源收集集合」**（`images/texts` 提取逻辑），与「能否拖放生成」无关 | `App.js:L4141` / `App.js:L5446` 定义处上下文为 `images: t, texts: n` 返回对象 |
| 节点可生成性取决于 spawnable Set | 节点可生成性取决于 `nodeTypes` 注册表 + `Z`(spawnNode) 入口；`textNode/audioNode/textConcatNode` 均在 `nodeTypes` 注册（L31121/L31128/L31137），**均可生成** | `App.js:L31117-L31145`（nodeTypes 含 27 项）|

**结论**：spawnable Set 是「从画布节点提取可下发资源」的过滤器，不是拖放白名单。旧文档据此质疑 `textNode/audioNode` 不可生成属于**误判**，已作废。

## 1. 运转图景（画布引擎全貌）

画布主体是一个 ReactFlow 实例（`Be` 组件），在 `App.js:L37120` 以 `<Be>` 渲染，挂载全部交互 handler 与 `nodeTypes`/`edgeTypes` 注册表。

端到端链路：

```
用户操作（拖放 / 右键 / 连线 / 拖拽停止）
   │
   ├─ onDrop: Lr (L36293)  ──→ 分类处理（模板/任务清单/本地文件/URL）
   │       └─ 本地文件 → ii(uploadFile, L1888) 落盘 canvas/drop → Z(imageNode...)
   │
   ├─ onConnect: xn (L31883) ──→ 批量连线 / 方向翻转 → Pe(edges)
   ├─ onConnectEnd: Fr (L36245) ──→ 拖到空白处 → 生成 ghostTarget 预览
   ├─ onPaneContextMenu: Cn (L31933) / onNodeContextMenu: wn (L31947)
   │       └─→ ve(contextMenu state) 打开右键菜单
   ├─ onNodeDragStop: jt (L31513) ──→ 落位 / 触发吸附
   └─ onPaneClick: An (L31993) ──→ 关闭菜单 / 取消选择
        │
        ▼
   节点生成入口 Z (L35536) 注入回调 → 渲染 nodeTypes[Z.type]
        │
        ▼
   节点内部回调（onGenerate=sr / onSpawnImageNode=Hn ...）驱动 AI 生成 / 资源同步
```

画布节点 ≠ 离散组件，而是**统一由 `Z` 注入 `data.*` 回调**后交给 `nodeTypes` 分发渲染（`App.js:L31117`）。

### 2.3 节点生成入口 `Z`（spawnNode）

定义于 `App.js:L35536`，是**所有节点生成的唯一统一入口**。行为：

1. `id = \`${e}-${Date.now()}\``（L35538）—— 以 type 前缀 + 时间戳生成唯一 id。
2. 按 type 计算默认 `style.width/height`（L35543-L35567）—— 不同节点尺寸不同（`promptNode` 420×420、`rhWebappNode` 820×720 等）。
3. **按 type 注入 `data.*` 回调**（L35568-L35605）—— 这是节点能力的来源：
   - `onGenerate: sr`（仅 `promptNode`）
   - `onGenerateText: cr`（仅 `textNode`）
   - `onGenerateVideo: ur` / `onGenerateSD2Video: dr` / `onGenerateDiscountVideo: pr`
   - `onGenerateCustom: _r` / `onAIAssist: gr`（仅 `customNode`）
   - `onSpawnImageNode: Hn`（`gridMergeNode`/`videoToGifNode`/`imageCompressNode`/`faceMosaicNode`/`compareNode`）
   - `onCrop: Fn`（`imageNode`/`promptNode`）
   - `onSplit*/onSplitOne*`（`gridSplitNode`）
   - `onCaptureToBox: Kn`（`director3dNode`/`panoramaNode`）
   - `onUploadAsset`（`sd2VideoNode`→`Yn` / `discountVideoNode`→`Xn`）
   - `aiAppApiUrl/aiAppApiKey`（`rhWebappNode`）
4. 返回节点对象交给 ReactFlow `G`（nodes state）。

> **关键**：新增节点类型必须①注册到 `nodeTypes`(L31117) ②在 `Z`(L35536) 注入所需回调，否则节点渲染或功能缺失。这与旧文档「只加 spawnable」的说法相反。

### 2.4 资源收集集合 `spawnable`（非白名单）

定义于 `App.js:L4141` 与 `App.js:L5446`（两处重复，内容一致）：

```
new Set([promptNode, imageNode, imageBoxNode, videoNode, sd2VideoNode,
         discountVideoNode, gridSplitNode, gridMergeNode, cropNode,
         urlToImageNode, fileToUrlNode, panoramaNode, videoExtractNode])
```

用途：在「收集画布可下发资源」逻辑中，判断某节点 type 是否属于「图片类」（`i.has(e.type)` 为真则归入 `images`，否则归入 `texts`）。`textNode/audioNode` 因不在集合内，其 `data.text` 走 `texts` 分支（L4144-L4153）。**与拖放生成无关**。

## 3. 关键数据流

### 3.1 拖放落盘（onDrop → uploadFile）

`Lr`(onDrop, L36293) 按 `dataTransfer` 内容分 4 路：

1. **模板**（`application/x-maomao-template`）：`JSON.parse` 后若是 `{id,name,graphData.nodes[]}` → `sn.current(e, t)` 载入工程（L36301-L36304）。
2. **任务清单**（`application/x-mutiwindow-task`）：解析 `url`+`mediaType` → 调 `Z` 生成对应节点（`discountVideoNode`/`audioPlayerNode`/`textNode`/`imageNode`），`mediaType` 决定 type（L36309-L36332）。
3. **本地文件**（`dataTransfer.files`）：按扩展名/MIME 分类（L36337-L36346）：
   - 文本 → `FileReader.readAsText` → `Z(textNode)`（L36351-L36360）。
   - 图片/音频/视频 → **`ii(e, {subfolder:'canvas/drop', preferThumbnail:l, ...})` 落盘**（L36363）→ 返回 `url` 走 http 则 `Z(imageNode/audioPlayerNode)`（L36369-L36378）；失败 fallback `readAsDataURL` base64（L36384-L36395）。
4. **纯文本/URL**（`text/plain`）：`http`/`data:image`/`blob:` → `Z(imageNode,{imageUrl})`，否则 `Z(textNode,{text})`（L36399-L36403）。

> **重要纠正（呼应模块2/模块3）**：画布级 `onDrop` 通过 `ii`(uploadFile, L1888) 把本地文件**落盘到 `canvas/drop` 子目录**（L36363-36368）。模块3 提到的「图片节点 `B`(onDrop L29188) 拖入 URL 只存内存不落盘」是**节点内部** onDrop 行为，与画布级 onDrop 不同——两者不矛盾，但旧文档未区分层级，易误导。

### 3.2 连线（onConnect → 批量 + 翻转）

`xn`(onConnect, L31883) 逻辑：

- 取 source/target 节点，计算各自 handle 的 `position`（left/right）（L31884-L31894）。
- 若 **同侧连线**（a===o）→ 提示「请连接节点的不同侧」并 return（L31895-L31898）。
- 若 source 在 left、target 在 right → **自动翻转**（交换 source/target，L31899-L31904）。
- 若 source 节点被选中且选中节点 >1 → **批量连线**：对每个选中节点 `Pe(edge, r)`（L31905-L31919）。
- 否则单条 `Pe(s, e)`（L31921-L31925）。

`kn`(onNodeClick, L31980) 也支持「多选后点目标节点」批量连线（L31980-L31991），与 `xn` 互补。

### 3.3 连线拖尾预览（onConnectEnd → ghostTarget）

`Fr`(onConnectEnd, L36245)：当 `!t.isValid && t.fromNode` 时，计算落点坐标 `we({x,y})`，清除旧 `ghost-target` 并追加一个 `type:'ghostTarget'` 的透明节点（L36257-L36269），用于预览「松手将生成新节点」的位置。`ghostTarget` 组件 `Nh` 已在 `nodeTypes` 注册（L31144）。

### 3.4 右键菜单（Cn/wn/En/On → ve）

- `Cn`(画布右键, L31933)：写 `ve({x,y,type:'canvas'})`。
- `wn`(节点右键, L31947)：写 `ve({x,y,type:'node',nodeId:t.id})`。
- `En`(选区级, L31958) / `On`(选区结束, L31968)：写 `ve({type:'selection'})`。
- 三者均先经 `Sn(e)` 判断是否在 INPUT/TEXTAREA 内（是则跳过，L31934/L31948/L31959）。
- 菜单内容由 `ve.type` 决定；`An`(onPaneClick, L31993) / `Ue` 负责关闭。

### 3.5 3D 导演台（Th / director3dNode）

`Th`(Director3DNode) 定义于 `App.js:L28388`，注册于 `nodeTypes`(L31141)。

- 通过 `Gt()` 取 `updateNodeData: i`（L28394），即 ReactFlow 的节点 data 更新器。
- 监听上游连线图片：`l = useHandle(handleType:'target')` → `u = 上游 source 节点 id`（L28399-L28402）→ 从上游节点 `data.imageUrl` 或 `imageBoxNode.data.images[activeIndex].url` 提取图片（L28403-L28417）。
- **截图落盘**（L28418-L28451）：`f`（onCapture 回调）遍历相机 captures，收集未同步截图 `n`，调 `a.onCaptureToBox?.(e, n)` 推入 imageBox（L28430）；若 `t.thumbnailDataUrl` 存在 → `ii(thumbnailDataUrl, {subfolder:'tasks', preferThumbnail:true})` **落盘到 tasks**（L28438-L28443）→ 成功后回写 `imageUrl/thumbnailUrl`（L28444-L28447）；失败保留 base64（L28448-L28450，warn `[Director3DNode] 缩略图 URL 化失败`）。

> 这再次印证模块2 的 `ii`(uploadFile, L1888) 落盘链：导演台截图 → `tasks` 子目录 → 回写 URL。

## 4. 存疑 Bug / 风险点

### P1-画布-01：spawnable 与 nodeTypes 不同步（已澄清，非 Bug）
旧文档误判 `textNode/audioNode/textConcatNode` 不可生成。实测 `nodeTypes`(L31121/L31128/L31137) 已注册，均可经 `Z` 生成。**建议**：删除旧文档相关质疑，避免后续维护者误删这三个节点的拖放能力。

### P2-画布-02：`onConnect` 批量连线依赖 `selected` 状态
`xn`(L31905) 仅在 `source 节点被选中且选中数>1` 时批量连线；若用户只选中 target 侧节点则不会批量。行为符合直觉但**未在 UI 提示**，新手可能困惑（无 Bug，体验风险）。

### P2-画布-03：`spawnable` 集合两处重复定义
`App.js:L4141` 与 `App.js:L5446` 完全重复同一 `new Set([...])`。若未来增删类型需改两处，易漏改导致资源收集不一致。**建议**：合并为单一常量（属 V1 优化，非红线禁止项）。

### P1-画布-04：节点内部 onDrop 不落盘（呼应模块3）
图片节点 `B`(onDrop, L29188) 拖入 URL 仅存内存（`data.imageUrl`），不经过 `ii` 落盘。与画布级 `Lr`(L36363) 的落盘行为形成「层级差异」。若用户期望拖入节点内的图也进本地库，当前不满足。**建议**：在节点内 onDrop 也调 `ii` 落盘（需确认是否红线允许改该点）。

### P2-画布-05：`Cn` 中 `n = e.clientX || e.clientX` 冗余
`App.js:L31938` `let n = e.clientX || e.clientX` 左右操作数相同，应为 `e.clientX || e.clientY` 或笔误。**无功能影响**（clientX 存在时恒为真），但属代码异味。

### P3-画布-06：`handleFollowLimit`/`edgeFxLimit` 硬阈值
`App.js:L37109-L37110` 设 `handleFollowLimit:60, edgeFxLimit:50`，节点/边超阈值时降级渲染（Lod 策略）。属性能保护，但大画布下视觉降级可能突兀（非 Bug）。

## 5. 审计结论

1. **画布引擎**由单个 ReactFlow 实例承载，全部交互 handler（`Lr/Ir/xn/Fr/Cn/wn/En/On/kn/jt/An/Yr`）在 `App.js:L37120` 与 `App.js:L37144` 之间一次性接线，定义点分散在 L31513-L36293。
2. **节点可生成性**由 `nodeTypes`(L31117, 27 项) + `Z`(spawnNode, L35536) 注入回调共同决定；旧文档所谓「spawnable 白名单缺失三节点」属误读，已纠正。
3. **落盘链路贯通**：画布级 `onDrop`(L36363) 与 3D 导演台截图(L28438) 均经 `ii`(uploadFile, L1888) 落盘，印证模块2 的本地数据层；唯一缺口是**节点内部 onDrop 不落盘**（P1-画布-04，呼应模块3）。
4. **批量连线**（`xn` L31905 / `kn` L31980）依赖选择集，无 UI 引导（P2-画布-02）。
