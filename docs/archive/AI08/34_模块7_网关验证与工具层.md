## ① 运转图景

两段职责不同但都属「非运行时支撑」：
- **`verify_gateway.py`**：三段离线测试（契约自验证 / webhook 重试 / 路由行为），用 `FakeClient` mock Lovart、不碰真实网络（`apimart-gateway/verify_gateway.py:L17-L22` 导入 `main` 但替换 `LovartClient`）。运行 `python3 verify_gateway.py`（`apimart-gateway/verify_gateway.py:L9`）。
- **`helpers.ts`**：localTool 所有路由共用的 HTTP 工具——JSON 响应（`json` L7）、错误（`sendError` L16）、JSON/原始 body 解析（`parseJsonBody` L20 / `readRawBody` L40）、multipart 解析（`parseMultipart` L53）、分页 DSL（`parsePagination` L128 / `buildPaginatedQuery` L165 / `paginatedResult` L237）。

两者都不直接服务前端请求，但决定了模块5/6 的行为契约是否成立。

## ③ 关键数据流与验证闭环（带 file:line）

### 流 A：网关契约离线验证（坐实模块5 结论）
- **audio 501 实锤**：`apimart-gateway/verify_gateway.py:L271-L276` 断言 `POST /v1/audio/speech` 与 `/v1/audio/transcriptions` 返 `501 + type=not_supported_error` → **独立确认模块5「音频/音乐 501 为噪音非 Bug」**。
- **信封形状**：`ok()`（`apimart-gateway/verify_gateway.py:L260`）、`err()`（`L262`）、`auth_error()`（`L266`）断言结构 → 与模块5 `lovart_to_apimart`（`main.py:L142`）产物结构一致。
- **模型映射防误配**：`resolve_prefer_models("kling-2.6")`→`kling_v2_6` 而非 `kling_v3`（`apimart-gateway/verify_gateway.py:L72-L76`）→ 关键边界，避免版本串错。
- **提交信封**：`/v1/images/generations` 返回 `{code:200, data:[{status:"submitted", task_id:"task_..."}]}`（`apimart-gateway/verify_gateway.py:L280-L285`）→ 与模块5 `_do_submit`（`main.py:L719`）写 `_TASK_META` 一致。

### 流 B：webhook 重试策略（坐实模块5 `_fire_webhook`）
- `_fire_webhook`（`main.py:L369`）在 `apimart-gateway/verify_gateway.py:L89` `_webhook_test` 离线验证：4xx 立即放弃（`L118-L120`）、5xx 最多重试 3 次（`L130-L132`，依赖 `WEBHOOK_MAX_RETRIES=3` `main.py:L39`）、200 标记完成（`L139`）。
- → **独立确认模块5 的 webhook 语义**，且验证套件把 `WEBHOOK_RETRY_INTERVAL` 置 0（`apimart-gateway/verify_gateway.py:L91`）以加速，不影响生产值（`main.py:L40`=10s）。

### 流 C：鉴权矩阵（坐实模块5 `OPEN_RELAY`/`USER_KEYS`）
- `apimart-gateway/verify_gateway.py:L191-L209` 四态断言：`OPEN_RELAY=false` 无 Bearer→401、未知 user_key→401 不 fallback、`OPEN_RELAY=true`→放行、已知 user_key 映射 AK/SK→200。
- → 与模块5 `resolve_client`（`main.py`，门5 提交路由调用）鉴权逻辑一致，确认无默认凭据 fallback 漏洞。

## ④ 存疑 Bug / 边界契约（对照 P0–P2）

### 边界契约
1. `verify_gateway.py` 为**离线断言**，不保证运行时 Lovart 行为；仅验证网关自身逻辑契约。
2. `localTool/src/utils/helpers.ts:parseMultipart` 无第三方依赖，但**不支持嵌套 multipart / 大文件流式**——超大上传可能内存爆（一次性 `Buffer.concat`，`localTool/src/utils/helpers.ts:L44`）；属已知简化。
3. `buildPaginatedQuery` `sortBy` 白名单外自动降级 `rowid`（`localTool/src/utils/helpers.ts:L218`），安全；但 `filters` DSL 的 `camelToSnake`（`localTool/src/utils/helpers.ts:L152`）仅做简单大写转下划线，**不校验列名合法性**——若前端发未定义 filter 键，会转成不存在的列名导致 SQL 报错（非注入，但可能 500）。

### 存疑项
- **P-本5（验证套件未覆盖 pending_confirmation 轮询）**：`verify_gateway.py` 的 `FakeClient.get_status` 直接返 `done`（`apimart-gateway/verify_gateway.py:L164`），**未模拟 `pending_confirmation` 中间态**。模块5 的 `AUTO_CONFIRM`/`pending_confirmation` 分支（`main.py:L823/L846`）缺乏独立离线验证 → 建议补 `FakeClient` 返回 `pending_confirmation` 用例，坐实模块5 该分支。
- **P-本6（验证套件未覆盖 confirm 路由）**：`/v1/tasks/{id}/confirm`（`main.py:L882`）在 `verify_gateway.py` 中未被断言 → 模块5 确认回路仅代码审计，缺离线测试。
- **P-本7（multipart 内存风险）**：`localTool/src/utils/helpers.ts:L44` `Buffer.concat` 全量缓冲，超大文件上传有 OOM 风险；建议流式写盘。
