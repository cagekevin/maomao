## 二、`resourceAdded` 双源与 extension 资源瞬态性（重要归因）

### 派发链
1. **Service Worker 采集**（`src/background.ts`）：右键"发送到资源"→ `handleSaveToTransit`（L65）→ 构造 `newResource`（L81，**`source:'extension'`**）→ `chrome.runtime.sendMessage({action:'resourceAdded',...})`（L102）。
2. **rescan 入库后**：localTool rescan(`Ev`) → 前端监听 `mutiwindow-task-completed`(L31428) → 资源经 `Sv`(L43552) 入库 → 面板刷新。

### 关键注释（L80）
> `// 直接使用原始 URL，不再经过本地引擎上传`

⇒ **扩展采集的资源 `source='extension'` 直接用原始 URL，不落盘到 localTool**。这解释了：
- 为什么 `TASKS P2 #7` rescan 孤儿清理只清 `source='local-tool'`（extension 资源本就在 DB 瞬态，不归 rescan 管）；
- 为什么 extension 资源刷新/离线即丢（与红线 §3.2「严禁 CDN 直链」精神一致——extension 源是已知瞬态设计，非 bug）；
- `background.ts:L91` 仅保留 storage.local 最多 5 条（防 OOM），App 侧用 localforage 接管。
