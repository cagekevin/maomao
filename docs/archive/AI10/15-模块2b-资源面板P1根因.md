# 模块2b · 资源面板「拖入/粘贴 URL 不落盘」根因（TASKS P1）

> 模块2（`11-模块2-本地数据层与Rescan.md`）指出 `Zr`@L1827 = uploadRemoteUrlToLocalTool 负责远程 URL 上传；交叉流（流二）指出采集流存在双数据源割裂（P1）。本篇深挖**资源面板拖入/粘贴 URL 后未落盘**的具体代码根因。
> 方法：grep `App.js` 资源面板回调 `z`@L29159 / `B`@L29188 / `text/plain` 分支坐实，行号以 2026-07-21 快照为准。
> 纯文档审计，未触源码。

---

## 1. 现象（TASKS P1 复述）

用户在资源面板**拖入网页图片 URL** 或 **粘贴图片链接** 后，资源短暂出现在面板（内存），但刷新/重开画布后**丢失**——即未落盘到 localTool（18080）。

---

## 2. 根因（grep 坐实）

资源面板的 URL 接收走两个 `useCallback`：

### 2.1 粘贴回调 `z` @L29159
```
z = Y.useCallback(async e => {            @L29159
  ...
  source: 'paste'                          @L29176 / @L29183
  ...
  // text/plain 分支：仅写入内存 transitItems
  text/plain                               @L29180 / @L29203
})
```
- 当粘贴内容为 `text/plain`（即裸 URL / 纯文本链接）时，分支只把条目塞进 `transitItems`（内存态），**未调用** `Sv()`(入库 @L42838) 也**未调用** `ii()`/`Zr()`(落盘)。

### 2.2 拖入回调 `B` @L29188
```
B = Y.useCallback(async e => {            @L29188
  ...
  // 仅当拖入的是 File（dataTransfer.files）才走 ii()/Zr()
  // 拖入的是 text/uri-list 或 text/plain（链接字符串）则同样只进内存
})
```

### 2.3 关键缺口
| 输入形态 | 是否落盘 | 原因 |
|----------|----------|------|
| 拖入本地 File | ✅ 落盘 | 走 `ii(e,{subfolder:'canvas/drop'})` / `Zr`@L1827 |
| 拖入/粘贴 **URL 字符串**（text/plain / text/uri-list） | ❌ 不落盘 | 分支只写内存 `transitItems`，缺 `Sv()`+`Zr()`/`ii()` 调用 |
| 采集流 resourceAdded | ⚠️ 部分 | 走 `Zr`@L1827 远程上传（见交叉流流二），但 id 体系与 rescan 不统一 |

> **根因结论**：资源面板对"URL 字符串"输入的处理分支**只更新内存 `transitItems`，缺少 `Sv()`(POST /api/resources/save) + `Zr()`(uploadRemoteUrlToLocalTool) 的落盘调用**，与"拖入本地文件即落盘"（模块4 流三）形成不一致。这正是 P1 描述的"拖入 URL 不落盘"的直接代码原因。

---

## 3. 修复方向（仅建议，不修改源码）

在 `z`@L29159 与 `B`@L29188 的 URL 字符串分支内，补一段：
1. 识别为 URL → 调 `Zr(url)`(@L1827) 下载到 `uploads/migrated/`（与采集流一致）；
2. 调 `Sv()`(@L42838) 入库 → `POST /api/resources/save`；
3. dispatch `mutiwindow-task-completed`(@L43640 等) 触发 rescan 刷新。

> 这样可统一"文件拖入"与"URL 拖入"两条路径，关闭 P1。注意：需复用模块2 已坐实的 `Zr`@L1827/`Sv`@L42838，不要新建上传逻辑（避免重复实现、违背单一出口原则）。

---

## 4. 与交叉流流二的关系

流二（采集流 `resourceAdded`）已用 `Zr`@L1827 远程上传，但资源面板粘贴/拖 URL 这条**交互路径**未复用该逻辑 → 形成 P1。两处合并修复可一并缓解交叉流 §6 列出的"双数据源 transitItems vs resources"风险。
