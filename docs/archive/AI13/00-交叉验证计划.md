# AI13 · 交叉验证工作区（11 份审计的互相校验 + 冲突实锤）

> 工作目录：`docs/AI13/`（唯一产出目录，不写 docs/ 根、不写 AI01~AI12、不改任何源码）
> 日期：2026-07-21
> 依据：`CLAUDE.md` 红线 + `docs/TASKS.md` 五关门3/门4（交叉仲裁）
> 代码权威：`src/_engine/App.js` + `config.js` + `localTool/` + `apimart-gateway/`

## 0. 任务定位（用户原话意图）

11 个 AI 各写各的模块审计（平行重复，互相隔离、互不读取）。用户要的不是第 12 份平行审计，
而是**用这 11 份互为参照做交叉验证**：以最细的 AI05 为基准，逐模块、逐文件拿其他 AI 的说法来比，
**遇到不一致才去源码 grep 实锤**，裁决谁对谁错，最终收敛出可信结论。

→ 本工作区只产出「比对表 / 冲突清单 / 裁决表」，不重写他们的内容，不另起审计。

## 1. 纪律边界（强制，对齐红线 §3）

- **只产文档**：不修改 `App.js`/`config.js`/`dist/`/vendor/映射表根文件/`localTool`/`网关`。
- **只交叉验证**：读 AI01~AI12 的审计 md，比对说法；冲突点 grep 源码核实。
- **裁决以代码为准**：任何冲突，回 `App.js` 实 grep，不靠 AI 自述投票。
- **不回写权威文档**：发现的错锚只记在 AI13 的裁决表里，待用户确认后由用户/上游回写。

## 2. 方法（用户指定：基准 + 逐文件比对 + 冲突实锤）

- **基准**：AI05（模块最细、18 个深一度文档，且已自纠 `canvas-state-change` 等错锚）。
- **流程**（每个模块重复）：
  1. 读 AI05 对应模块文件，提取「断言清单」（每个混淆名/行号/事件/数据流结论）。
  2. 读其他 AI 的同模块文件，逐条比对：✅一致 / ❌冲突 / ➖缺失。
  3. 对 ❌冲突点，grep `App.js`/`config.js`/`localTool` 实锤，写进裁决表（带 `file:line` 证据）。
  4. 输出本模块的《模块N-交叉比对.md》+ 把冲突裁决汇总进《裁决表.md》。
