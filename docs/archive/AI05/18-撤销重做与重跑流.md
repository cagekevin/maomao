## 1. 撤销/重做机制（实证，与 ARCHITECTURE §4.5 一致）

### 1.1 store 状态（L21217-21225）

### 1.2 核心写入函数 `r`（L21580-21606）
```21580:21606:src/_engine/App.js
function r(t, n = {}) {
  let { trackUndo: r = true, persist: i = true } = n;
  e(e => {
    let n = e, a = Eu(n), o = t(n);
    if (ed(a, xu(o))) return {...o, undoStack: r?n.undoStack:o.undoStack, ...};  // 无变化不入栈
    let s = r && n.undoBatchDepth > 0 && n.undoBatchSnapshot === null,
        c = r && n.undoBatchDepth === 0 ? td([...n.undoStack, a]) : o.undoStack,  // 批量中不入栈，结束才入
        l = {...o, undoStack: c, undoBatchSnapshot: s?a:o.undoBatchSnapshot, ...};
    return i && Su(xu(l)), l;   // 持久化到 localforage
  });
}
```
- `Eu`=快照当前状态；`ed`=JSON 相等判定(L21569)；`td`=压栈并截断超 `Xl`(最大深度)（L21572）；`Su`=持久化(L21178)；`xu`=去函数引用(L21216 附近)。
- **关键**：任何状态变更经 `r` 走「快照→判等→压栈→持久化」，**不直接调 setNodes/setEdges 改 ReactFlow 内部态**，而是改中央 store 再驱动渲染。这与 05 节「onConnect 直接 setEdges」不冲突：onConnect 改的是 ReactFlow 的 edges state（L37125 接线），而 store 的 `r` 负责可撤销的画布数据（nodes/edges 业务数据）。
- **无变化不入栈**（L21589 `ed(a,xu(o))` 早退）：避免无意义撤销步。

### 1.3 undo 执行（L22304-22313）
```22304:22313:src/_engine/App.js
undo: () => {
  let n = t(), r = n.undoStack[n.undoStack.length - 1];
  r && (e({...Tu(r), clipboard:n.clipboard, clipboardPasteCount:n.clipboardPasteCount, undoStack:n.undoStack.slice(0,-1)}), Su(r));

```
- 弹出栈顶快照 → `Tu`(L21216 还原，补回函数引用) → 应用 + 持久化 `Su(r)`。
- 快捷键：`Ctrl/Cmd+Z` undo、`Ctrl/Cmd+Shift+Z` 或 `Ctrl+Y` redo(L28319)。

### 1.4 批量撤销（beginUndoBatch/endUndoBatch L21615-21645）
- `beginUndoBatch` 加深 `undoBatchDepth` 并拍快照；`endUndoBatch` 在 depth 归零时，若批次内有变更则把**批次起始快照**压栈（L21639）——整批操作算一步撤销。
- 用于拖拽/多节点操作合并成单步。

### 1.5 持久化 `Su`（L21178-21215）
- `fu()` 取 localforage 实例 → 写画布状态（含 undoStack 一并持久化，刷新后可继续撤销）。

> **结论**：ARCHITECTURE §4.5「撤销/重做直接 setNodes/setEdges」表述**不够精确**——实际是中央 store `r` 统一接管（快照+压栈+持久化），ReactFlow 的 `setNodes/setEdges` 仅在接线层(L37125)用于即时交互，业务数据变更经 store。建议 ARCHITECTURE §4.5 补「store.r 统一撤销栈，Su 持久化」。

## 2. mutiwindow-rerun-task 重跑流（实证）
