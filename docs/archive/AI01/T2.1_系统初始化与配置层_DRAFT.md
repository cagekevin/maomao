## 一、运转图景（这个模块负责什么）

系统启动后，前端扩展（`src/main.tsx` → lazy `App.js`）先装配配置层（`config.js`）与存储编排器（`Q`=StorageManager），再完成「本地模式」登录伪装（`Oa` 永远返回本地 token），随后建立与 localTool(:18080) / 网关(:9004) 的连接，最后恢复画布状态、拉取资源。

本模块是**所有其它模块的底座**：鉴权头（`Jh`）、API 基址解析（`tr`/`ar`）、存储读写（`Q` 及其后端 `wr`/`Nr`/`Pr`/`jr`）、画布状态键（`Cr`）都出自这里。改任何上层功能前，若涉及「存哪 / 走哪个地址 / 带什么鉴权头」，都要回到本模块。

## 三、关键数据流（本模块发出的接缝）

### 流1：去登录（启动即"已登录"）
```
Oa()(L3543) → 永远 'local-mode-token'
  → Ba()(L3637) 恒 true
  → Jh()(L30712) 返回 {Authorization:'Bearer '+token}
  → 所有 API 调用带此头
```
注意：`ka`(L3547)/`Aa`(L3552) 改的是 localStorage + Q.setConfig，不影响 `Oa` 的兜底返回值（localStorage 无则仍返 'local-mode-token'）。故「重置配置」清不掉"已登录"——属预期。

### 流2：存储读写路由（StorageManager 选后端）
```
Q.getConfig/X(L1441 附近)
  → jr()(L1345) 缓存层
    → 资源键(kr,L1339 命中 Or 前缀)? 走 wr(localTool,L1297) : 走 wr/localStorage
  → 画布状态: Pr(localforage,L1446) / Cr(L1294) 构造键
```
边界：若 localTool 未启动，`wr.isAvailable=false`，`getKV` 打「❌ window.localTool 不存在」日志（L1302），功能受限不崩（PROJECT_LOG 已记）。

### 流3：API 基址解析（双服务桥接）
```
ar(e,t)(L863): 传入址 e / 官方址 $n
  → localhost:3000 且官方非本地 → 回退网关 n
  → 否则用传入址 r
tr(e)(L846): 补 /api 后缀
```
`Hr = localEngineBase()`(L1732) 与 `vv = LOCAL_ENGINE.base`(L42808) 均指向 18080；`ot='http://127.0.0.1:9004'`(L43084) 为网关。跨设备/改 host 时 P2「统一 base URL」风险在此。

## 五、存疑 Bug / 待确认（交后续 AI 对抗审计）

1. **`ar` 双地址逻辑是否仍被调用**：`ar`(L863) 含 `localhost:3000` dev 演示分支（`DEV_DEMO_SITE`，config.js L65），但当前 USE_LOCAL_ENGINE=true 主走 localTool，`ar` 实际在哪些调用点生效需 grep 确认（疑似仅官方 API 兼容路径，本地模式可能未被触达）。**待核实**：避免误以为改 `ar` 会影响本地模式请求。
2. **`Oa` 兜底 token 与 `ka` 保存的 token 关系**：`Oa` 优先读 localStorage(Ca)，无则返 'local-mode-token'。若用户曾 `ka` 存过真 token，「重置配置」`Aa`(L3552) 清 localStorage 后 `Oa` 仍返兜底——登录态永不掉。属设计还是副作用？待确认。
3. **`jr` 缓存 TTL 600s（Dr,L1337）对配置热更新影响**：改 config.js 后若经 `jr` 读旧键，600s 内可能读缓存。需确认高频改的配置键是否绕开 `jr`。
4. **`Cr` 画布键与 localforage 版本**：`Z.CANVAS_STATE_PREFIX='canvas-state-v1-'`(func-mapping)，换项目用 `Cr(projectId)` 区分；多项目并存时 localforage 键冲突面待查。

> 以上存疑项均**未改代码**，仅供后续审计/修复参考。本模块未触碰红线禁区（仅读 `App.js`/`config.js`，未改 `dist/`/vendor/原版残留）。
