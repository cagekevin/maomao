## 二、与本地引擎/落盘耦合

- **缩略图落盘**：`ii(t.thumbnailDataUrl, {subfolder:'tasks', preferThumbnail:true, ...})`（L28438）。
  `ii` 是上传落盘核心（模块2 已坐实 L1888），走 localTool :18080 `/api/files/upload`。
- **失败退化**：catch 仅 `console.warn`，保留 base64（内存）。即 localTool 离线/失败时，缩略图不落盘、只存节点 data 的 base64 —— 刷新后若未再落盘则仍 base64，**不阻塞但弱耦合**（与模块2 P2「依赖 18080 在线」同源）。

## 三、潜在 Bug / 存疑（交后续 AI 对抗审计）

1. **P2 · 缩略图落盘失败静默**（L28448）：`ii` 抛错仅 warn，节点 imageUrl 停留在 base64（未转 http URL）。
   影响：localTool 偶发不可用时 3D 导演台缩略图不持久化为 URL，画布重渲染可能重复 base64。
   修复建议：失败时不覆写 imageUrl（当前逻辑已如此：try 内才 `i(e,{imageUrl:n.url})`，故 base64 保留是预期退化，非硬 bug，仅记录）。
2. **`wh` 编辑器组件已展开**：见 `T2.10_wh与In图片盒子深挖_DRAFT.md`。实测 `wh`(L28276–28387) 局部符号干净（`r`/`i`/`a`/`o`/`s`/`c`/`n`/`e`/`t`），**无模块级 R/B/we 冲突**；原"重名高发指向 wh"担忧被推翻，真正重名高发区是画布主组件（README 问题4）。
3. **`onCaptureToBox`(`Kn`) 已展开**：`Kn`(L32367) 是薄包装 → `In.current`(L32009 `useRef`→L32488 `=qn`)；`qn`(L32370) 是真正 push 实现，把 3D 截图 dataURL 推入 imageBoxNode，未落盘（P1 同源，见 T2.10）。
