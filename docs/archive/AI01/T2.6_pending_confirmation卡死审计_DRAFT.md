# T2.6 · pending_confirmation 卡死链审计（_DRAFT · 未验证）

> 续 `符号冲突深度核查_DRAFT.md` 问题4「待后续审计确认」项。
> 目标：确认 `pending_confirmation` 是否真为死锁 bug，即前端是否暴露确认 UI 并调用 `requestConfirm`。
> 行号快照 2026-07-21，会漂移，以 grep 为准。归属 V1。未改任何源码。

## 结论（高置信）

**是真实死锁 bug。** 前端定义了确认适配器 `makeConfirmAdapter`/`requestConfirm`（L32960–32979）与
节点标记函数 `markNeedsConfirm`（L32980–32985），但**全文件没有任何一处调用 `requestConfirm`，
也没有任何 UI 组件读取 `confirmTaskId`/`confirmGatewayBase`/`confirmKind`/`confirmToken` 来渲染确认按钮**。

一旦网关返回 `error.code==='pending_confirmation'`，节点被置 `await_confirm` 后轮询 `return` 终止，
且没有任何代码路径可以把它从 `await_confirm` 推回轮询或完成时状态 —— 永久卡死。

---

## 证据链（逐条 grep 坐实）

### E1. 确认函数"只定义不调用"
- `requestConfirm` / `makeConfirmAdapter`：grep 全文仅 L32960–32979 定义处命中，**0 处调用**。
- `makeConfirmAdapter` 产出 `confirmAdapter`（闭包），`requestConfirm(confirmAdapter, taskId)` 调它；
  二者皆无第二引用点。

### E2. 标记属性"只写不读"
- `confirmTaskId` / `confirmGatewayBase` / `confirmKind` / `confirmToken`：grep 全文**仅 L32983 一处**（写入），
  无任何 UI 组件读取这几个属性。
- 即：`markNeedsConfirm` 把状态写到节点上，但没有任何渲染分支据此显示"确认"按钮。

### E3. 检测与终止（来自符号冲突核查问题4）
- `resolveNeedsConfirm`(L32957)：`return taskInfo?.error?.code === 'pending_confirmation' ? taskInfo.id : null`
- 三处命中触发终止：
  - L33059（图 image）：`cid = resolveNeedsConfirm(taskInfo); if(cid){ markNeedsConfirm(z, cid, 'image', R, h); return }`
  - L33546（视频 video）：`markNeedsConfirm(z, cid, 'video', _, m); return`
  - L34218（视频 video）：`markNeedsConfirm(z, cid, 'video', g, m); return`
- `markNeedsConfirm`(L32980) 把节点置 `status:'await_confirm'`，轮询函数 `return` 不再落盘、不再轮询。

### E4. 网关侧触发条件
- `apimart-gateway/main.py` L34：`AUTO_CONFIRM = os.getenv("AUTO_CONFIRM","true").lower()=="true"` 默认 true。
- L177 注释：AUTO_CONFIRM=false 时把待确认状态转成结构化错误，附带 `task_id` 让调用方确认。
- 即：仅当网关 `/v1/tasks` 返回体带 `error.code==='pending_confirmation'` 时前端才会进入死锁；
  默认 `AUTO_CONFIRM=true` 下网关通常不返回此 code，故线上罕见，但**一旦 Lovart 侧强制确认即必卡**。

## 死锁状态机

```
生图轮询中
   └─ 网关返回 error.code === 'pending_confirmation'
         └─ resolveNeedsConfirm() 命中 → 返回 taskId
               └─ markNeedsConfirm() → 节点 status = 'await_confirm'（写入 confirmTaskId 等）
                     └─ 轮询函数 return（终止）
                           └─ 期待 UI：读 confirmTaskId → 渲染确认按钮 → 调 requestConfirm(confirmAdapter, cid)
                                 ❌ 但：requestConfirm 全文件 0 调用；confirmTaskId 全文件 0 读取
                                       → 无按钮、无回调 → 永久 await_confirm（卡死）
```

## 修复建议（仅提方案，不执行；修须单独立项 + 红线 §3.3 第10条首句确认）

1. **最简止血**：在 `await_confirm` 状态节点渲染"确认/重试"按钮，点击调用 `requestConfirm(makeConfirmAdapter(gatewayBase, authToken), confirmTaskId)`，成功后把节点 `status` 改回轮询中并重启轮询。
   - 需要拿到 `gatewayBase`/`authToken`：当前 `confirmGatewayBase`/`confirmToken` 已写入节点，UI 可直接读。
2. **网关侧兜底**：保持 `AUTO_CONFIRM=true` 默认，或在 L177 路径返回可重试结构，避免强制 pending。
3. **防抖/超时**：`await_confirm` 节点加超时自动转失败提示，避免静默永久卡。
