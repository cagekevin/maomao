# 12 - 多窗口通信与生成调度审计

> 审计范围：跨窗口/跨 iframe 通信机制（mutiwindow-* CustomEvent + localStorage 剪贴板）、生成调度中枢（ar 故障转移、sr/cr 入口、Jn/tr 核心）、全局快捷键。
> 方法论：五道闸门。所有锚点经 grep 复核（见 `校验报告-AI11.md`）。

## 12.1 多窗口通信机制

本扩展以"画布主窗 + 插件端/本地端"多窗口架构运行。跨窗口不依赖 `chrome.runtime` 长连接，而是用 **`window.dispatchEvent(new CustomEvent('mutiwindow-*'))` + `localStorage` 共享** 实现（同源多文档/iframe 可见）。

### 事件清单（grep 全量）

| 事件名 | 生产方（锚点） | 消费方（锚点） | 用途 |
|--------|---------------|---------------|------|
| `mutiwindow-task-completed` | `App.js L38481`（AI轮询完成）、`L43640`/`L43676`/`L43697`（资源落盘完成）、`L44406`（统一同步后） | `App.js L31428`（监听→触发 `Ev()` rescan 入库 L31426） | 任务完成→后端 rescan 统一入库 |
| `mutiwindow-images` | `App.js L16369`/`L16497`（复制图片到剪贴板） | `App.js L36001`（粘贴为 imageNode） | 图片跨窗口粘贴 |
| `mutiwindow-clipboard` | `App.js L16376`/`L16503`（localStorage 兜底） | `App.js L36034`（CustomEvent 缺失时回退读取） | 图片剪贴板持久化兜底 |
| `mutiwindow-rerun-task` | `App.js L44343` | `App.js L36618`（监听重跑） | 重跑任务 |
| `mutiwindow-update-task-meta` | `App.js L41032` | `App.js L43764`（监听更新元信息） | 任务元数据更新 |
| `mutiwindow-open-builtin-settings` | `L4883`/`L4922`/`L5828`/`L5867`/`L9518`/`L26385` | `App.js L43774`（监听打开设置） | 跨窗口打开内置设置 |
| `mutiwindow-open-schedule-settings` | `L4883`/`L5828`/`L26348` | `App.js L43774` | 跨窗口打开排程设置 |

### 关键结论
1. **完成事件驱动 rescan**：`mutiwindow-task-completed` 是资源统一入库的触发器（L31426 调 `Ev()`=rescan@L42838）。模块2已确认 `Ev` 全量扫库落盘——即所有生成完成都走同一入库路径，设计一致。
2. **图片剪贴板双通道**：优先 CustomEvent `mutiwindow-images`，监听方未就绪时回退 `localStorage('mutiwindow-clipboard')`（L36034）。**注意拼写**：代码全局用 `mutiwindow`（少一个 l），非 `multiwindow`——文档若写 `multiwindow` 均错。
3. **设置面板跨端唤起**：插件端点 "打开设置" 经 CustomEvent 通知本地端画布打开对应面板（L43774 统一注册两个 open 事件）。

## 12.2 生成调度中枢（同名遮蔽高发区）

### 身份表（本轮新增，补入 07）
| 名字 | 模块级 | 组件局部（主画布 useCallback） | 说明 |
|------|--------|-------------------------------|------|
| `tr` | `App.js L846`（vendor 封装） | `App.js L34350`（文本生成核心） | **同名遮蔽** |
| `ar` | `App.js L863`（vendor 封装） | `App.js L34759`（多模型故障转移调度器） | **同名遮蔽** |
| `Jn` | `App.js L89`（**SVG Logo 图标组件**，非生成！） | `App.js L32490`（图片生成核心） | **同名遮蔽（第四处身份）** |

> 重要纠正：模块3/07 把 `Jn` 记为 "LogoIcon vs generateMedia"。精确事实：`function Jn({size})`(@L89) 是一个 SVG 图标组件（渲染品牌 Logo path，L92-100），与生成无关；`let Jn = Y.useCallback(...)`(@L32490) 才是图片生成核心。`Jn` 实为"图标组件"与"生成函数"两个完全无关的身份。

### 调度链路
```
sr (图片入口, L34786)
  └─ or(r) 查模型调度器 (L34782) → 若有 enabled 调度配置
       └─ ar(e,o, r=>Jn(e,t,n,r,i,1))  (L34789, 故障转移)
  └─ 否则直接 Jn(e,t,n,r,i,a)  (L34792)

cr (文本入口, L34794)
  └─ or(r) → ar(e,i, r=>tr(e,t,n,r))  (L34797)
  └─ 否则直接 tr(e,t,n,r)  (L34802)

lr (chat 分支, L34804) → zc(`${o}/v1/chat/completions`, ...)  (L34815, 文本大模型直连)
```

### `ar` 故障转移调度器（`App.js L34759`）
- `rr.current`（`Set`）防同一任务并发重入（L34760/L34779）。
- `va(t)` 取候选模型列表（L34762），逐个 `n(a)` 尝试（L34766）。
- `ir(e)` 检查是否已成功（L34767），成功即返回（L34769）；`mg(o)` 命中中断信号则停（L34772）。
- 全部失败才报 "全部 N 次尝试均失败"（L34776）。

### `Jn` 图片生成核心（`App.js L32490`）
- 校验 key → 置 `loading:true`（L32513-32518）→ 调网关 `${R}/v1/images/generations`(L32796，模块3已确认) → 轮询 `${R}/v1/tasks/${taskId}`(L33005) → 结果 `ii(u,{subfolder:'tasks'})`(L33049) 落盘 → dispatch `mutiwindow-task-completed`(L38481)。
- **文本分支不在此**：文本走 `tr`/`lr`。

### `tr` 文本生成核心（`App.js L34350`）
- 校验文本模型 key（`Xi(o)` L34353）→ 选 endpoint `s?l:t`（L34367，即 `C` 文本端点或默认）→ 调文本大模型。
- 失败回写 `loading:false`（L34361）。与图片 `Jn` 结构平行但端点/落盘不同（文本结果不必然走 `ii`，见 12.3）。

## 12.3 文本/音频生成落盘合规性（红线 §3.2 复核扩展）

模块3仅验证了**图片**生成 `ii(u,{subfolder:'tasks'})`(L33049) 落盘合规。本轮扩展验证文本/音频：

- **音频转写**：`App.js L12830` `${s}/v1/audio/transcriptions` —— 走配置 endpoint `s`（非 CDN 直连），符合红线。
- **文本 chat**：`lr`(L34804) → `zc(.../v1/chat/completions)`(L34815) 直连用户配置文本端点。文本结果通常直接写 node data，**不经 localTool 文件化**（文本无 URL），不违反 §3.2（§3.2 针对"媒体结果需落 localTool"，文本本身非媒体文件）。
- **结论**：图片/音频媒体结果经 `ii` 落 18080；纯文本结果留 node data。红线 §3.2 对文本分支无强制文件化要求，现行设计合规。

## 12.5 对抗审计（Gate4）

1. **`Jn` 双重身份混淆风险（高）**：`Jn`@L89 是本地定义的 SVG Logo 图标组件（**非 vendor 导入**，vendor `Jn` 已改名 `I`），`Jn`@L32490 是图片生成核心。任何文档/读者将 L89 当作生成函数会完全错锚。**已记入 07 同名遮蔽总表（已更正 vendor 误述）**。
2. **多模型故障转移的静默中断**：`ar`(L34772) 用 `mg(o)` 判断中断，若 `ir(e)` 返回非标准中断串但任务实际失败，调度会继续下一模型——符合预期重试，但 `mg` 的匹配规则（L31152 起 `mg` 函数）若过于宽松可能误判成功。建议（不改代码）核查 `mg` 匹配集。
3. **`mutiwindow-clipboard` localStorage 无限增长**：图片 base64 写入 `localStorage`(L16376/L16503)，大图可能超出 localStorage 5MB 配额导致 `setItem` 抛错（被 try 吞掉，L16503 上下文）。粘贴大图时剪贴板兜底可能失效——中风险，记 T。
4. **完成事件重复触发 rescan**：`mutiwindow-task-completed` 在图片(L38481)、资源落盘(L43640 等)、同步(L44406) 多处 dispatch，每处都会被 L31428 监听触发一次 `Ev()` rescan。密集完成时 rescan 频繁全量扫库，有性能开销但功能正确。
