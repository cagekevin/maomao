## 10.1 持久化 API 三层结构

画布状态持久化由 `StorageManager`（`Q`，定义在 `App.js L1476`）承载，底层四后端在模块 1 已映射，本轮用代码精确修正其身份：

| 变量 | 真实后端 | 锚点 | 说明 |
|------|----------|------|------|
| `wr` | localTool WASM 引擎（`window.localTool`） | `App.js L1297` | `get/set/setObject/remove` 全走 `window.localTool.getKV/saveKV` |
| `Mr` | chrome.storage.local | `App.js L1364` | 仅 MV3 扩展环境可用（`typeof chrome < 'u'`） |
| `Nr` | window.localStorage | `App.js L1418` | 永远可用 |
| `Pr` | localforage（`Sr.default`） | `App.js L1446` | **注意：模块1文档曾把 `Pr` 误记为 chrome.storage，实际 `Pr`=localforage，`Mr`=chrome.storage** |

`Q.getStatus()`（`App.js L1480`）返回 `{localTool, chromeStorage, localStorage}` 三后端可用性；`getAvailableEngines()`（`App.js L1487`）据此组装引擎列表。

### 修正声明（覆盖模块1）
`docs/AI11/02-模块1-初始化与配置层.md` 中把 `Pr` 标为 chrome.storage 是错误的，正确映射为：`Pr`=localforage，`Mr`=chrome.storage。已记入 `07-文档纠错建议.md` 的 Name-shadowing/变量纠正表，待用户授权回写权威文档。

## 10.2 画布状态保存/加载函数

| 函数 | 锚点 | 行为 |
|------|------|------|
| `Q.saveCanvasState(e,t)` | `App.js L1642` | `Cr(e)` 规范化 key → `setLocalforage(n,t)` 双写 |
| `Q.loadCanvasState(e)` | `App.js L1646` | `Cr(e)` → `getLocalforage(t)` |
| `Q.saveCanvasStateWithVersion(e,t,n)` | `App.js L1650` | 带版本号防冲突保存 |
| `Q.setLocalforage(e,t)` | `App.js L1636` | **双写**：`wr.isAvailable() && wr.setObject(e,t)` + `Pr.set(e,t)`（localforage） |
| `Q.getLocalforage(e)` | `App.js L1627` | `wr.get(e)` 优先，回退 `Pr.get(e)` |

### 版本冲突保护（L1650-1664）
- 空画布（`!t.nodes || t.nodes.length===0`）直接跳过保存（`App.js L1652`）。
- 远程版本 `o > n` 时拒绝覆盖并告警（`App.js L1659`）。
- key 规范化用 `Cr(e)`（`App.js L1643/1647/1651`），与 `Z.CANVAS_STATE_PREFIX`（`Q.setObject` 内 `App.js L1529`）配合拒绝空画布落库。

### 调用点（主画布保存流）
主画布在 `onNodesChange`/`onEdgesChange` 触发保存，调用 `Q.saveCanvasStateWithVersion(en.current, r, t)`（`App.js L31728`），`en.current` 为画布 ID，`t=Date.now()` 作版本号。保存失败/冲突处理在 `App.js L31729-31740`：
- 冲突时回读 `Q.getObject(Cr(en.current))`（`App.js L31732`）并强制刷新内存节点（`Qt.current`/`W([...e])`，`App.js L31738`）。

## 10.3 节点字段白名单序列化（工作流导出）

工作流"导出为应用"功能用两个 `Set` 做字段过滤：

| 变量 | 锚点 | 含义 |
|------|------|------|
| `ov` = nodeDataFieldSet | `App.js L42001` | 允许序列化的 data 字段白名单（prompt/text/imageUrl/model… 共 ~50 个） |
| `sv` = nodeCallbackFieldSet | `App.js L42002` | 节点回调/状态字段黑名单（id/label/onGenerate… 这些不导出为输入） |

### 序列化逻辑 `fv(e)`（`App.js L42032`）
- 先 `dv(e)`（`App.js L42013`）做深拷贝（JSON stringify 去函数，`App.js L42019`）。
- 遍历每个节点的 `data` 条目（`App.js L42039`），过滤规则 `App.js L42040`：
  ```
  if (sv.has(a) || a.startsWith('on') || !ov.has(a) || a === 'text' || !uv(o) && !Array.isArray(o)) continue;
  ```
  即：**跳过回调字段、以 `on` 开头的字段、不在白名单 `ov` 的字段、非基础类型且非数组的字段**。
- 生成输入字段描述对象（id/key/label/type/required/defaultValue/sourcePath/group），`App.js L42045-42058`。
- 最后按 `id` 去重（`App.js L42061`）。

### 应用快照 `pv(e)`（`App.js L42064`）
`dv(e.workflowGraph)` 取图 → 与 `selected` 输入字段组装 `{appId, projectId, appName, workflowSnapshot, ...}`（`App.js L42067` 起）。`workflowSnapshot` 即经 `dv` 净化的图。

## 10.5 对抗审计（Gate4）

1. **双写一致性风险**：`setLocalforage`（`App.js L1636`）先写 localTool 再写 localforage，若 localTool 不可用（`wr.isAvailable()` 为 false）则只写 localforage——此时 `getLocalforage`（`App.js L1627`）先读 `wr.get`（返回 null，因未连接）再回退 `Pr.get`。**结论：localTool 离线时数据仍在 localforage，不会丢，但两后端可能短暂不一致。** 非 bug，属预期降级。
2. **版本号来源**：保存用 `Date.now()`（`App.js L31723/31728`），多标签页并发保存时若两标签页同毫秒保存，版本号相等，`o > n` 不成立（`App.js L1659`），后者覆盖前者，无冲突保护。低概率竞态，记 T（待观察）。
3. **序列化丢字段风险**：`fv` 的 `uv(o)`（`App.js L42011`）只接受 string/number/boolean 或数组，嵌套对象字段（如 `data.config`）若不在 `sv` 黑名单且不在 `ov` 白名单，会被 `!ov.has(a)` 跳过（`App.js L42040`）。即**复杂配置对象默认不导出**，符合白名单最小化原则，但自定义节点若依赖 `ov` 未列出的字段会静默丢失。
