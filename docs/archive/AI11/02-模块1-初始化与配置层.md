# 02 · 模块1 系统初始化与配置层（四段式审计）

> 锚点：Oa/Ba/Jh/ar/tr/Nr/Pr/jr/Cr + config.js；全部 grep 坐实于 2026-07-21。
> 边界契约为本模块与模块2/3/4 的接缝。

## 一、运转图景（这个模块做什么）

系统启动：`src/main.tsx` lazy 加载 `src/_engine/App.js`（用 `v2/components/ErrorBoundary` 包裹）→ App.js 顶层 import 从 `config.js` 与 vendor 包拉入全局配置/符号 → 组件挂载后走 bootstrap 流程：检测 localTool 连接（轮询 `${Hr}/api/status`）→ 加载 config（`Q`/StorageManager）→ 恢复项目/画布状态 → 注册事件监听 → 进入画布。

配置层（`config.js`）是纯导出模块，**不依赖运行期状态**，是模块2/3 解析 base URL 的唯一来源。

## 四、关键数据流（本模块参与的部分）

1. **base URL 解析**：模块2/3 用 `vv`/`Hr`（`http://127.0.0.1:18080`，var-mapping L117–118）访问 localTool；`R`/gatewayBase 用 `config.DEFAULT_ENDPOINT`=9004 访问网关。注意 `USE_LOCAL_ENGINE=false` 时文件请求走 9004 无上传路由（TASKS P2-5，ARCHITECTURE X3.3）。
2. **鉴权**：`Oa()` 读 `auth_token` → `Jh()` 注入 `Authorization` 头；本地模式 `Ba()` 恒 true（去登录改造后）。

## 六、存疑 Bug（锚点待复核，门4 留痕）

- **host 硬编码 18080**：`Hr`/`vv`/`U_` 均为字面量 `http://127.0.0.1:18080`（var-mapping L117/118/233），未读 `config.LOCAL_ENGINE.base`。若改 config 端口不生效（TASKS P0 残留）。
- **中文路径 Latin1 乱码**：localTool 侧落盘文件名编码问题，前端无直接锚点，归模块2/后端。
- **`Q` vs `me` 重命名**：行1 `Q as me`，App.js 内 `Q` 实为 vendor 的 StorageManager；grep `function Q` 不存在。审计引用 `StorageManager` 时须用 `Q`(混淆名) 并注明「即 vendor 导入的 StorageManager」。

## 七、对抗审计质询（自问自答）

- Q：`Oa()` 读的是哪个存储？A：`localStorage.getItem(Ca)`，Ca=`auth_token`（var-mapping L102）；非 `Q`/StorageManager，与「不绕过 StorageManager」红线不冲突（auth_token 是明文 localStorage 约定，原版保留）。
- Q：`USE_LOCAL_ENGINE` 改 false 会怎样？A：文件请求走 `REMOTE_BASE`=9004，但网关无 `/api/files/upload` 路由 → 上传失败（TASKS P2-5 已记）。
