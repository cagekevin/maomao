## 一、bootstrap 启动初始化流（ARCHITECTURE X6 落地）

```
src/main.tsx → lazy App.js → ReactFlow 挂载
  → useLocalTool hook (L19096) 建立连接检测：checkConnection + 轮询 ${Hr}/api/status
  → StorageManager.syncAllToLocalTool() (L1608)：把 USERS/MEMBERSHIP/PROJECTS/GLOBAL_TASKS/
      CUSTOM_NODE_TEMPLATES/APP_SETTINGS/TRANSIT_RESOURCES/TRANSIT_GRID_COLS 从
      chrome.storage(Mr)/localStorage(Nr)/localforage(Pr) 同步到 localTool(wr)
  → 恢复项目 → localforage(Pr) 恢复画布状态(canvas-state-v1-{id}) → 注册事件监听
```
> 关键：`syncToLocalTool`(L1591) 按后端优先级 Mr(chrome.storage)→Nr(localStorage)→Pr(localforage) 读取，写入 `wr`(localToolBackend)。**不绕过 StorageManager**（符合红线 §3.3）。
> ⚠️ **后端身份更正（见 10/11）**：`Mr`=chrome.storage.local、`Nr`=window.localStorage、`Pr`=localforage、`wr`=localTool WASM。本段旧标注曾误将 Mr/Nr 颠倒，已校正。
> ⚠️ **覆盖方向风险提示**：`syncToLocalTool`(L1591) 逻辑为「`if (Mr.isAvailable()) { let t = await Mr.get(e); if (t !== null) return ... await wr.set(e,t); }`」——即**仅当 chrome.storage 有值才覆盖 localTool**。若 localTool 数据比 chrome.storage 新，重启后会被旧 chrome.storage 覆盖（见 11 §11.5 风险3，P 级）。本处原"符合红线"指"走 StorageManager 不绕过"，与 11 的"覆盖方向风险"不矛盾，但措辞需区分。

## 六、存疑 Bug（门4 留痕）

- **bootstrap 时序**：`syncAllToLocalTool` 依赖 `wr.isAvailable()`（localTool 已连）。若 localTool 未启动，`wr.isAvailable()` false → 同步静默失败（L1592 直接 return），不报错但数据未落盘。属已知限制（18080 连不上功能受限不崩，TASKS）。
- **`Se.current` 全局锁跨 effect**：仅防统一同步 effect 自身重入；若 `mutiwindow-task-completed` 高频派发 + 监听 L31426 触发 `Ev()` rescan，与统一同步并行无共享锁，理论上 rescan 与 upload 可能交错，但 rescan 只读不写任务，风险低。
- **AI 生成落盘(`ii`) vs 统一同步落盘(hook uploadFile) 双路径**：同一结果 URL 可能被两处各落盘一次（生成完成先 `ii` 落盘，统一同步再遇未本地化又落盘）。但 `ii` 落盘后 URL 已变 `/files/tasks/...`，统一同步 L44369 跳过 → 不重复。✅ 逻辑自洽。
