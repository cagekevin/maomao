## 13.1 资源面板 CRUD 与后端路径

资源 store 集中在 `App.js L42838` 起，全部走 HTTP（localTool/gateway），无直连 CDN：

| 函数 | 锚点 | 调用端点 | 说明 |
|------|------|----------|------|
| `Sv`(saveResource) | `App.js L42838` | `POST ${vv}/api/resources/save` | 保存资源（含 favorite） |
| `Cv`(setFavorite) | `App.js L42851` | 复用 `Sv` + `isFavorite` | 收藏切换 |
| `wv`(deleteResource) | `App.js L42857` | `POST ${vv}/api/resources/delete?id=` | **DB only**（模块2已记：不删磁盘文件，孤儿源） |
| `Tv`(clearResources) | `App.js L42866` | `POST ${vv}/api/resources/clear` | 清空（按 folder） |
| `Ev`(rescan) | `App.js L42883` | `POST ${vv}/api/resources/rescan` | 全量扫盘入库（模块2/12 已确认） |

> `vv` 为资源后端 base（与 `R` 网关 base 同源或同一 localTool 端口）。所有资源操作经后端，符合红线 §3.2。

### 资源项删除 UI
资源列表项 `onDelete` 回调链：`App.js L40512`（调度组项，但同模式）→ `L41014`/`L41940` 资源项 → 最终调 `wv`(L42857)。删除仅发后端 delete 请求，**不清理磁盘文件**（与模块2 TASKS P1-2 一致）。

### 资源面板右键/上下文
- 网页图片"发送到资源"引导文案：`App.js L382`（`在网页图片/视频上点击右键 选择"发送到资源"`）。
- 资源面板的 `resourceAdded` 事件监听：`App.js L43527`（`e.action === 'resourceAdded' && e.resource`），用于实时刷新列表。该事件由模块3/4 的 `ii`(L33049 等) 落盘后 dispatch（模块3已确认）。

## 13.2 工作流导出/发布流

### 入口与组装
发布面板在 `App.js L42480` 起校验：空画布(`!p.nodes.length`)→拒绝(L42482)、未选启动参数(`E===0`)→拒绝(L42486)。通过后：

```
await Dg(`/workflow-apps/publish`, pv({
  projectId, projectName, workflowGraph: p,
  appName: h, description: _, visibility: y, appId: x, inputFields: C
}))          // App.js L42492
```

- `pv`（`App.js L42064`，模块3/10 已确认）：`dv(e.workflowGraph)` 净化图（`App.js L42065`）+ `e.inputFields.filter(selected)`（L42066）→ 组装 `{appId, projectId, projectName, appName, description, visibility, workflowSnapshot, ...}`。
- `workflowSnapshot` 即经 `dv` 去函数深拷贝的图；`inputFields` 由 `fv`(L42032) 从节点白名单 `ov`/`sv` 提取（模块10已确认）。

### `Dg` 封装链
`Dg`（`App.js L38287`）= `Tg(e, {method:'POST', body:t})` → `Tg`（`App.js L38232` 统一 HTTP 封装）→ `Sg(e)` 拼完整 URL（`App.js L38242`）→ `wg(c,l,i)` 实际 fetch（`App.js L38255`）。
- `Tg` 自动 JSON 序列化非 FormData/Blob body（`App.js L38254`），401 触发 `Aa()` 鉴权（`L38256`）。
- 发布目标 `/workflow-apps/publish` 走 **网关/后端**（非 CDN），合规。

### 发布结果处理
`App.js L42502-42506`：`e.success` 判断 → `r.data ?? r` → 回调 `a?.(i)` 传回 appId。失败仅 `console.error` + 提示（`L42507`）。

## 13.3 拖拽落盘一致性（URL/文件 → 资源/节点）

拖拽入口分布在多处，关键对照：

| 源 | 锚点 | 落盘行为 | 合规 |
|----|------|----------|------|
| 画布 `onDrop`(Lr) | `App.js L36243`/`L29446` | 节点拖入画布 | — |
| 资源面板文件拖入 | `App.js L29193` | `R(e.dataTransfer.files)` 读文件 → 走 `ii`/`Zr` 落 localTool | ✅ |
| 网页 URL 拖入资源 | `App.js L29156`/`L29203` | `text/plain`/`text/uri-list` → `Zr(url,{subfolder:'migrated'})` | ✅（模块2已确认） |
| 节点间拖拽（layer/puzzle） | `App.js L7699`/`L8315` | `application/x-yimao-*` 内部移动，不落盘 | — |

### 一致性结论
- **文件/URL → 资源**：均经 `Zr`(L1827，上传远程 URL/文件到 localTool) 或 `ii`(L1888) 落盘，符合红线 §3.2。
- **已知缺口（模块2已记）**：资源面板"URL 直接输入/拖入"部分路径不落盘（@L29176 类），与 rescan 不清理 extension 孤儿叠加产生僵尸记录。本轮复核 `L29193`/`L29203` 确认**文件拖入与 uri-list 拖入均落盘**，缺口仅在纯文本 URL 输入框分支。

## 13.5 对抗审计（Gate4）

1. **发布失败不回滚**：`Dg('/workflow-apps/publish')` 失败时仅提示（`L42507`），若后端已部分写入（如 resource 已存但 app 记录失败）无回滚。低概率，记 T。
2. **`workflowSnapshot` 字段丢失风险**：`pv` 依赖 `ov`/`sv` 白名单（模块10 L42001/L42002）。自定义节点若用 `ov` 未列字段，发布后该字段在 `workflowSnapshot` 中缺失——与序列化同理，属预期最小化但可能丢配置。已在模块10记。
3. **资源删除孤儿（高危，P1-2 已知）**：`wv`(L42857) 仅删 DB 记录不删磁盘文件，rescan 只清理 `source==='local-tool'` 孤儿（模块2 `resources.ts L120`）。extension 来源资源删除后磁盘残留 + rescan 不清理 = 僵尸文件。模块2已建议，待授权回写 TASKS.md。
4. **`vv` 与 `R` 双 base 一致性**：资源操作走 `vv`，AI/发布走 `R`（`Sg` 拼）。两者应指向同一 localTool/网关，若配置漂移会导致资源存 A、生成读 B。建议核查 `vv` 定值（grep 未直接命中赋值，疑为 `R` 别名或 config 派生），记 T 待深挖。
