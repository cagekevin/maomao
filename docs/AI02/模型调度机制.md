## 一、这是什么

maomao 是基于 React + Vite 的前端画布 / AI 工作流应用。每个生成类节点（绘图 / 文生文 / 视频 / sd2Video / 折扣视频等）都要选一个"模型"去调用 API。

节点的 `selectedModel` 有两种形态：

1. **普通模型选择**：`selectedModel` 是一个具体模型名（来自该节点类型对应的配置列表，如 `drawingModel` / `textModel` / `videoModel` / `sd2VideoModel` / `discountVideoModel`，这些是换行分隔的模型名字符串）。
2. **模型调度（schedule）**：`selectedModel` 可为 `schedule:<id>` 形式的引用，指向一条"调度配置"。调度配置把多个模型按 steps（每步 `model` + `retries`）排成一条**失败重试 / 故障转移序列**，运行期按顺序逐个尝试，某个模型失败就自动切下一个。

一句话：**一条调度 = 一个按序尝试的"故障转移 / 重试"序列**。调度只决定"尝试哪些模型、什么顺序、每个试几次"；最终每个模型能否真正调用，仍受模型权益（entitlements）门控（见第六章）。

调度配置由 `src/services/modelSchedules.js` 管理，存于 localStorage 键 `modelSchedules`（常量 `ta`，见第二章），并随 GAS 云同步（见第七章）。

## 二、存储与数据结构

### 2.1 存储键一致性（铁律）

三处指向同一字符串 `'modelSchedules'`，不可写错：

`src/config/constants.js` L15：
```text
var ta = `modelSchedules`;
```

`src/config/storageKeys.js` L11：
```text
MODEL_SCHEDULES: `modelSchedules`
```

`src/services/modelSchedules.js` L2 引用 `ta`，保证读写同源：
```text
import { ta } from '../config/constants.js';
```

调度配置实际落在两处：
- **浏览器 localStorage** 键 `modelSchedules`（`la` 读 / `ua` 写）；
- **存储层 `Q`（本地引擎 KV）** 同键（`da` 经 `Q.setObject(ta, e)` 落库，见 2.4）。

### 2.2 一条调度对象的字段

校验函数 `oa`（`src/services/modelSchedules.js` L28-44）产出标准结构：
```text
{
  id:        string,              // sa() 生成 "sch_<time36>_<rand36>"
  name:      string,              // 默认 "未命名调度"
  category:  'text'|'image'|'video',  // 非法值回落 'image'
  enabled:   boolean,
  steps:     [{ model: string, retries: number }],  // 经 aa() 校验
  createdAt: number,
  updatedAt: number
}
```

### 2.3 `schedule:<id>` 引用约定

- 构造：`ga(e)`（`modelSchedules.js` L108-110）→ `` `${ra}${e}` ``，其中 `ra = 'schedule:'`（L6）。
- 解析：`_a(e)`（L111-113）→ `e.startsWith('schedule:') ? e.slice(9) : null`，取冒号后 9 个字符之后的 id。
- 节点 `selectedModel` 存的就是 `ga(id)` 的结果；配置面板列表项选中时同样写入 `ga(t.id)`（见 L2959、L2964-2966）。

### 2.4 steps 校验规则（上限由 `aa` / `ia` 实现）

`aa`（`modelSchedules.js` L11-27）逐条规整 steps：
- `ia`（`modelSchedules.js` L7-10）：单步 `retries` 归一化到 **1–3** 闭区间；
- 最多 **5 步**：`if (t.length >= 5) break;`（L16）；
- 总 `retries` ≤ **10**：累加 `n`，超出时裁剪当前步并在满是 10 时 `break`（L20-21）；
- 空模型名跳过（L17-18），最终 `oa` 在 `r.length === 0` 时返回 `null`（L33）拒绝非法调度。

UI 侧同样有硬约束提示（`App.js` L37861）：
```text
生成时按顺序逐个尝试，直到成功。同一模型最多重试 3 次，最多 5 个模型，总调度不超过 10 次。
```
UI 禁用逻辑：`l.length >= 5 || x >= 10` 时禁止再加模型（`App.js` L37744）；`retries` 步进按钮 disabled 于 1 / 3（`App.js` L37829、L37842）。

## 五、运行期调度逻辑（核心 · 故障转移 / 重试）

### 5.1 解析器 `or`（只取 enabled 调度）

`src/App.js` L32355-32358：
```text
or = Y.useCallback(e => {
  let t = _a(e);
  return t && la().find(e => e.id === t && e.enabled) || null;
}, [])
```
`_a(e)` 取 id，`la()` 现读 localStorage，只匹配 `enabled` 的调度。解析失败或非调度则返回 `null`，走普通模型名路径。

### 5.2 扁平化 `va`（按 retries 重复成有序数组，上限 10）

`src/services/modelSchedules.js` L114-121：
```text
function va(e) {
  let t = [];
  for (let n of e.steps) for (let e = 0; e < n.retries; e++) {
    if (t.length >= 10) return t;
    t.push(n.model);
  }
  return t;
}
```
例：`[{m:'A',retries:2},{m:'B',retries:1}]` → `['A','A','B']`。上限 10 与 2.4 校验一致。

### 5.3 调度循环 `ar`（真正的故障转移）

`src/App.js` L32332-32354：
```text
ar = Y.useCallback(async (e, t, n) => {
  if (rr.current.has(e)) return;          // 同一节点防重入
  rr.current.add(e);
  let r = va(t);                           // 扁平化成有序模型数组
  try {
    for (let i = 0; i < r.length; i++) {
      let a = r[i];
      M(`调度「${t.name}」：尝试 ${i + 1}/${r.length} · ${a}`), await n(a);  // n(a)=真正执行（如 Jn/tr/er...）
      let o = await ir(e);                 // 等节点结果
      if (!o) { M(`调度成功：${a}`); return; }            // 成功即止
      if (mg(o)) { M(`调度已中断：${o}`); return; }       // 用户中断则停
      i < r.length - 1 ? M(`${a} 失败，正在切换下一个模型…`)
                       : M(`调度结束：全部 ${r.length} 次尝试均失败`);
    }
  } finally {
    rr.current.delete(e);
  }
}, [M, ir])
```
`ir(e)`（L32316-32331）轮询等节点 `loading` 结束并返回 `errorMsg`（`null` 成功、非空失败、`mg(o)` 视为中断、`'调度超时'` 兜底）。

**机制结论**：这就是"故障转移 / 重试"——按顺序 `await n(a)` 调用每个模型，失败（或超时）自动切下一个，命中成功即返回，全部失败则报"全部 N 次尝试均失败"。

### 5.4 五个生成节点的接入

每个节点执行器先 `or(model)` 解析，命中就包一层 `ar` 调度，否则直连：

批量运行（"全部运行"/画布执行）同样透传 `selectedModel`，非调度时回退该类型默认模型首项（`App.js` L33036-33040）。

## 六、与模型权益（modelEntitlements）的衔接

`src/services/modelEntitlements.js` 决定"单个模型是否可用"，与调度的"顺序/重试"正交叠加：

- 调度只编排**尝试顺序与次数**（`va` + `ar`），不判断模型是否有权限。
- 真正执行每一步时，执行器内部仍走权益门控。例如文本执行 `lr`（`App.js` L32377-32387）会 `Xi(i)` 区分内置/三方，再用 `Ai(i)` 取 deny reason，无权限则 `throw Error`。绘图 `sr`→`Jn`、视频 `er`/`Qn`/`$n` 同理受权益约束。
- 顶层放行 `ki`（`modelEntitlements.js` L26-48）：本地模式 `LOCAL_MODE_ALLOW_ALL_MODELS` 时直接返回 `{ access: 'allowed' }`（L29-31）；否则查 `bi.entitlements[t]` 或在线模型列表 `xi`，全无则 `denied`（L44-47）。

**叠加效果**：调度把"A 失败换 B"，但若 B 也因权益不足被 `lr`/`Jn` 抛错，该步同样判失败并继续切下一个；权益足够才成功退出循环。

## 七、订阅、跨标签页与云同步

### 7.1 本地订阅 + 跨标签页实时刷新 `ha`

`src/services/modelSchedules.js` L96-107：
```text
function ha(e) {
  let t = t => { e(t.detail ?? la()); },          // 监听 modelSchedules:change 自定义事件
      n = t => { t.key === 'modelSchedules' && e(la()); };  // 监听 storage 事件（跨标签页）
  window.addEventListener(na, t);                 // na = 'modelSchedules:change'
  window.addEventListener('storage', n);
  return () => {                                  // 返回清理函数，避免泄漏
    window.removeEventListener(na, t);
    window.removeEventListener('storage', n);
  };
}
```
- `na`（`modelSchedules.js` L5）= `'modelSchedules:change'`，由 `ua`（L62-73）在写 localStorage 时 `dispatchEvent` 派发，detail 为新列表。
- `storage` 事件实现**跨标签页同步**：另一标签页改了 `modelSchedules`，本页 `la()` 重读刷新。
- 订阅点：`App.js` L2116（绘图节点 `ae`）、L37872（设置列表 `C_`）；清理函数均随组件卸载执行，未泄漏。

### 7.2 GAS 云同步（推送 `la()`、拉取合并 `Sa`）

- **启动加载** `xa`（`modelSchedules.js` L131-137）：从存储层 `Q.getObject(ta)` 读后 `ua(ya(e))` 载入 localStorage。`App.js` L40270 在启动流程 `xa()` 调用。
- **推送到云** `syncToCloud`（`App.js` L41058-41063）：
  ```text
  for (let t of ['app_settings','api_configs','users','membership','projects',
                 'presetPrompts','customNodeTemplates','modelSchedules',
                 'cloud_storage_config','local_templates']) {
    let n = t === 'modelSchedules' ? la() : ...   // ← 推送用 la() 现读 localStorage
  ```
  即 `modelSchedules` 分支直接取 `la()` 的本地列表上云。
- **拉取合并** `Sa`（`modelSchedules.js` L138-141）：
  ```text
  async function Sa(e) {
    let t = ba(la(), e);          // ba：按 id 去重合并（云端覆盖本地同名）
    return Array.isArray(t) && t.length > 0 && da(t), t;
  }
  ```
  pull 分支（`App.js` L41129-41131）：
  ```text
  if (e === 'modelSchedules') { await Sa(n); r++; }
  ```
  `ba`（`modelSchedules.js` L125-130）用 `Map` 按 `id` 合并：本地与云端同 id 以**云端**为准（`ya` 先本地后云端，后者覆盖前者），保证去重且不丢本地独有项。合并后经 `da` 同时写 localStorage + 存储层 `Q`，并触发 `ua` 派发 `modelSchedules:change` 全页刷新。

**结论**：调度配置可跨设备同步——本机编辑经 `syncToCloud` 上云，其他设备 `pull` 时 `Sa` 按 id 合并落库，启动期 `xa` 再载入。

## 八、本地模式

调度是**纯本地配置**（localStorage + 存储层 `Q`，随云同步但不依赖官方账号）。本地模式下：
- `modelEntitlements.js` 的 `LOCAL_MODE_ALLOW_ALL_MODELS` 为真时，`ki` 直接放行一切模型（`modelEntitlements.js` L29-31、L101 `Mi` 进入本地模式亦置该标志为 true）；
- 因此调度序列里的每个模型在本地模式都能直接 `await n(a)` 调用，不受在线权益额度限制；
- 调度本身（顺序、retries、enabled）与权益无关，只受 2.4 的本地校验约束，官方账号状态不影响调度编排。

## 十、改码前必查

1. **键名一致性**：`ta`（constants.js L15）、`MODEL_SCHEDULES`（storageKeys.js L11）、`localStorage` 键、`Q` 键必须同为 `'modelSchedules'`。改任一处需同步其余。
2. **`schedule:` 前缀编解码**：`ga` 拼 `schedule:`、`_a` 用 `slice(9)` 截取（前缀正好 9 字符）。改前缀长度必须同步 `_a` 的 `9`，否则解析错位。
3. **retries 上限**：`ia` 归一化 1–3，`aa` 限 5 步且总 ≤10，`va` 扁平化上限 10。三处上限需保持一致（UI 也限 L37744 / L37829 / L37842）。
4. **GAS 合并按 id 去重**：`ba` 用 `Map` 合并，同 id 云端覆盖本地。改合并策略勿丢本地独有调度；`Sa` 合并后必须 `da`（写 localStorage + `Q` + 派发事件）。
5. **推送用 `la()`**：`syncToCloud` 的 `modelSchedules` 分支必须 `la()` 现读，不能用内存旧值，否则云上不是最新。
6. **事件订阅清理**：`ha` 返回清理函数，`App.js` 两处 `useEffect` 均已 `return` 清理，新增订阅点务必在卸载时 `removeEventListener`，避免泄漏 / 重复刷新。
7. **enabled 过滤**：选择器只展示 `enabled && category` 匹配的调度（绘图 L2115、设置列表用全量）；运行期 `or` 也只取 `enabled`。改启用逻辑须保证两处一致，否则"已停用的调度"仍可能被选中却解析为 `null`。
8. **本地模式放行**：调度序列每步最终受 entitlements 门控；本地模式 `LOCAL_MODE_ALLOW_ALL_MODELS` 才全放行，勿在调度层绕过权益检查。
