## 二、Uc 拆分执行步骤（严格按顺序，禁止跳步）

### 步骤 1：在 App.js 插入 import
在现有 services import 之后（即 `import { yv, bv, xv, Sv, Cv, wv, Tv, Ev } from './services/resourceStore.js';` 这一行**之后**）新增一行：
```js
import { Uc } from './hooks/useLocalTool.js';
```
> 注意：App.js 是 CRLF 行尾。用 `replace_in_file` 时 old_str 取文件原样（含 CRLF），new_str 同风格追加，不要混 LF。

### 步骤 2：从 App.js 删除 Uc 整块（核心，用精确字符串）
用**一次** `replace_in_file` 删除从 `var Vc = 5e3,` 到 Uc 结束 `}` 的整段。
- old_str 起点：`var Vc = 5e3,`（上一行是 `  });` 即某组件闭合，是锚点）
- old_str 终点：Uc 的 `}`（`var Wc = Y.memo(({` 之前那一行）
- new_str：空字符串（删除），但**必须保留 `var Wc = Y.memo(({` 及其前的 `  });`**，即 old_str 不含 `var Wc` 那一行。

> ⚠️ 关键陷阱（已踩坑）：Uc 块前是 `  });` `}` 这类组件闭合 `}`。删除时**只删 Uc 自身那一段**，绝不能把前面组件的闭合 `}` 一并删掉，否则会留下孤立 `}` 或吃掉组件闭合，导致后续 JSX 被 rollup 误判为 "invalid JS syntax"。
> 做法：old_str 的**第一行必须是 `var Vc = 5e3,`**（不是前面的 `}`），最后一行是 Uc 的 `}`。这样前面的组件闭合 `}` 完全不动。

### 步骤 3：导出块补 `Uc`
在 App.js 的 `export {` 块中，找到 `yv, bv, xv, Sv, Cv, wv, Tv, Ev,` 这一行，在其后新增一行：
```
  Uc,
```
（保持前面 2 空格缩进 + 逗号风格，与块内其他行一致。）

## 六、血泪踩坑记录（本版 SOP 的由来，务必读）

执行阶段 4 时曾 2 次失败，根因相同：**用 `node` 脚本按行号 `splice` 删除大块代码**。

1. **行号偏移**：前序删除（阶段1-3）已让目标行号相对基线偏移数百行；插入 import 又让行号再变。基于"搜索出来的行号"做 `splice(17609,144)` 会**吃错行**——实际把 Uc 前组件的闭合 `}` 一并删了，或漏删导致残留孤立 `}`。
2. **CRLF 陷阱**：文件是 CRLF，正则 `/,$/` 匹配不到行尾逗号（逗号后是 `\r`），导致 `H_ = Y.memo(B_)` 行尾逗号没改成 `;`，造成连续赋值链断裂。
3. **孤立 `}` 的隐蔽性**：删块后若多/少一个 `}`，`node --check` 在 .js 文件上会把 ESM `export` 报成 "Unexpected token 'export'"（**这是 node 不识 ESM 的假阳性，不是真错**）；而 rollup/vite 会报 "invalid JS syntax / JSX" —— 后者才是真结构破损信号。两者不要混为一谈。
4. **唯一可靠做法**：大块删除一律用 `replace_in_file` + **精确字符串锚点**（如以 `var Vc = 5e3,` 为起点、Uc 的 `}` 为终点），绝不依赖行号。小块文本修正（如行尾逗号）也要先 `read_file` 确认文件真实字符（含 `\r`）再改。

> 结论：接手 AI 若发现 App.js 已破损，**不要就地用行号补 `}`**——直接 `git checkout src/App.js` 回到阶段3干净态，按本 SOP §二 精确字符串重做。
