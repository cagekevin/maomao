## 1. 分级原则

| 级别 | 判定条件 | 能否改 | 约束 |
|------|----------|--------|------|
| A 类 | 模块 `export {}` 中导出、且被 `App.js` L5–24 import 引用的名字 | 可改，但**必须同步改 App.js 对应 import 项** | App.js 是红线文件，改其 import 需 build 验证；建议解耦收尾后统一处理 |
| B 类 | 模块 `export {}` 中导出、但**未被 App.js import**（仅模块间互引或暂未被消费） | 可改，同步改所有消费方 | 先 grep 确认消费方清单 |
| C 类 | 模块内定义、未出现在 `export {}` 的私有变量/函数 | ✅ 最安全，直接改 | 改后 build 验证即可 |
| 禁区 | `App.js` 内部的短名（`ii`/`Xr`/`Vr` 等） | ❌ 禁止 | CLAUDE.md §3.1 红线 |

## 3. C 类（模块内私有，可安全改名）探查方法

每个模块的 `export {}` 块**以外的**函数/变量即 C 类。例如：

- `auth.js`：`export{}` 之外定义的 `Ga/Wa/Ka` 等辅助函数（如 `Ka(e)` 在 L134 定义、`Ga()` 在 L136 被引用）属 C 类，可改名。
- `localToolClient.js`：`export{}` 之外、L169 调用的 `Xr`（注意：此 `Xr` 是模块内定义、恰与导出 `Xr` 同名，需 grep 区分）等内部 helper。
- `modelSchedules.js`、`urlTools.js`：同理，凡未列入 `export{}` 的均为 C 类。

> **实操**：改名前对目标符号在模块文件内 grep，确认不出现在本模块 `export{}` 且不在 App.js/其他模块 import 中出现，即为 C 类。

## 4. 改名执行顺序建议（边解耦边改）

1. **优先 C 类**：每切出一个新模块，趁上下文清晰，立即语义化其模块内私有变量（B/C 类），改完 `npm run build` 验证。
2. **暂缓 A 类**：被 App.js import 的导出名，等全部解耦完成、确认无遗漏引用后，统一做一次"A 类改名 + 同步改 App.js import"的组合提交。
3. **重点排雷**：`localToolClient.js` 的导出名 `Vr/Kr/Xr/Zr/Qr/ri/ii/ai` 本就是从 App.js 原样剪切，App.js 经 re-export 块（约 L430–437）仍绑定这些名；改名 = 必须同步改该 re-export 块 + 全仓 grep 排除与 App.js 内部 storage 符号撞名。
4. **entry.js 依赖约束**：`urlTools.js` / `localToolClient.js` / `auth.js` 从 `entry.js` 取 base/React 时，只能用**合法第 4 导出 `t as Hn`**（或 `r as Vn` 同源），**严禁 import entry.js 内部 `Vn/Hn` 局部变量**，否则触发 TDZ（`Cannot access before initialization`）。改名若涉及这些模块的 entry 引用，须保持合法入口不变。
5. **禁区不动**：App.js 内部短名、`CLAUDE.md` §3.1 红线、`PROJECT_LOG.md` 记录的已知噪音（9004 404 / RootErrorBoundary useState null）一律不碰。

## 5. 验证门

- 每次改名后必跑 `npm run build`。
- A 类改名须额外确认 App.js import 项（及 re-export 块）已同步、且无 `Cannot access before initialization` / `is not a function` 类运行时错误（参考 `HANDOFF.md` 回退教训）。
- 改名遵循奥卡姆剃刀：无明确可读性收益不改，避免为改而改引入风险。
