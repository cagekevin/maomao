## 0. 为什么要拆

App.js 有 4.4 万行。改一个功能需要在 4.4 万行里翻，这就是问题。

拆的目的：**降低认知负担**。将来改网关请求，只看 `gatewayClient.js`（几百行）；改模型权益，只看 `modelEntitlements.js`（280 行）；改某个节点，只看那个节点的文件（几十到几百行）。

每次剪切后问自己：调用方是不是比之前更容易找到和理解这段代码了？是 → 有价值。不是 → 不拆。

## 2. 铁律

### 2.1 奥卡姆剃刀

每次动手前问：**"删掉这个改动有影响吗？"** 答不上来就不做。

具体禁止：
- ❌ 创建包装层（已有 `taskStore.js` 导出 `q_`、`J_`，不需要再包一层 `localTool.tasks.save()`）
- ❌ 创建桶文件（两个文件不需要 `index.js` 重导出）
- ❌ 为了一致性把 3 行函数单独建文件
- ❌ 给同一个东西起两个名字

### 2.2 改名策略（切和改分离）

**剪切时不改名**。把代码从 App.js 原样搬到新文件，保证行为不变。这一步只做一件事：改变代码的物理位置。

**切完后，逐个评估是否可以改名**。评估标准：
1. 100% 确定语义（`func-mapping.txt` + grep 函数体坐实）
2. 引用范围可控（只在新文件和 App.js 的 import 中出现，没有动态字符串引用）
3. 改后认知负担明显降低（`saveTask` 比 `J_` 更容易理解）

满足全部三个条件 → 改。有一条不满足 → 不改。

### 2.3 通用铁律

- 每步 `npm run build` 必须通过
- 从 App.js 中删除已切出的代码，不做两份保留
- 小步提交
- 不碰 `entry.js`、`vendor/`、`dist/`、`.css`

## 3. 拆什么（按依赖从底层到上层）

```
App.js 剩余内容 (~4.4 万行)
│
├── 阶段一：底层（无 UI 依赖，被上层大量调用）
│   ├── 模型权益/目录逻辑      (~280 行)  →  src/services/modelEntitlements.js
│   └── 网关请求（9004 fetch）  (~2000 行) →  src/services/gatewayClient.js
│
├── 阶段二：中层（依赖阶段一）
│   └── Zustand editorStore     (~5000 行) →  src/store/editorStore.js
│
└── 阶段三：上层（依赖阶段一+二）
    ├── React Flow 画布容器     (~3000 行) →  src/components/canvas/
    ├── 五 Tab 容器 + 面板      (~3000 行) →  src/components/panels/
    ├── 27 种节点组件           (~20000 行) →  src/components/nodes/
    └── 其他 UI 组件            (~10000 行) →  src/components/
```

## 4. 阶段一：切出底层

### 4.1 模型权益/目录

切到 `src/services/modelEntitlements.js`。

来源：App.js L31-311（约 280 行）。包含模型分类 `yi`、模型目录 `bi`、权益查询 `ki`、权益拉取 `ji`、订阅通知 `Di`/`Ti` 等。

**价值**：这些函数被模型选择器、节点组件大量调用。切出后找模型权益逻辑只看 280 行文件，不用在 4.4 万行里搜。

**安全**：纯函数 + 模块级变量，无 DOM 依赖，无 React 依赖。剪切风险极低。

步骤：
1. 创建 `src/services/modelEntitlements.js`
2. 从 App.js 剪切 L31-311 到新文件，添加需要的 import
3. App.js 改为 `import { yi, bi, xi, Di, Oi, ki, Ai, ji, Mi, Ni, Pi, Fi, Ii, Li, Ri, Si, Ci, wi, Ti, Ei } from './services/modelEntitlements.js'`
4. 删除 App.js 中已切出的代码
5. `npm run build` 验证

### 4.2 网关请求

切到 `src/services/gatewayClient.js`。

**价值**：目前 9004 的 fetch 散落在 App.js 各处——生图在 L32490 附近、生视频在其他地方、chat 又在另一个地方。切到一个文件后，所有网关请求的位置统一了，改网关逻辑只需要看这一个文件。

**安全**：纯剪切 fetch 调用，不改变请求/响应处理逻辑。

步骤：
1. grep App.js 中所有 `fetch` 到 9004 的调用点，列出完整清单
2. 按端点分组，逐个切到 `gatewayClient.js`
3. 每切一个端点就 build 一次

已知调用点（需 grep 确认精确行号）：
- `Jn`@L32490 — 生图主回调 → `POST /v1/images/generations` + 轮询 `GET /v1/tasks/{id}`
- `er`/`Qn`/`$n` — 生视频分派 → `POST /v1/videos/generations`
- `POST /v1/chat/completions` — SSE 流式 chat
- `POST /v1/images/edits` — 图编辑
- `POST /v1/uploads/images` — 图片上传到网关
- `GET /v1/balance` — 余额查询

## 5. 阶段二：切出状态管理

### 5.1 editorStore

切到 `src/store/editorStore.js`。

**价值**：最高。App.js 的核心骨架就是 Zustand store，约 5000 行。切出后 App.js 的复杂度直接降一个数量级。

**安全**：中等风险。store 与 React Flow 的 `onNodesChange`/`onEdgesChange` 深度耦合，剪切时需要精确定位 store 的边界。

关键混淆名（`var-mapping.txt` 已坐实）：
- `$` = editorStore（Zustand create 工厂）
- `W` = setNodes、`G` = setEdges
- `Qt` = nodesArray、`$t` = edgesArray
- `K` = contextMenu、`ve` = setContextMenu、`Ue` = closeMenuSetter
- `Et` = multiConnectMode、`Dt` = setMultiConnectMode
- `ov` = nodeDataFieldSet（序列化白名单）、`sv` = nodeCallbackFieldSet（不序列化）
- `Su` = saveEditorState、`xu` = extractPersistedState

步骤：
1. 定位 `$` 的 Zustand `create()` 完整起止行
2. 切到 `src/store/editorStore.js`，补必要的 import
3. App.js 改为 `import { $, W, G, ... } from './store/editorStore.js'`
4. 删除 App.js 中已切出的代码
5. build + 画布交互测试（添加节点、连线、拖拽、撤销）
