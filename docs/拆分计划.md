## 2. 铁律

### 2.1 奥卡姆剃刀

每次动手前问：**"删掉这个改动有影响吗？"** 答不上来就不做。

具体禁止：
- ❌ 创建包装层（已有 `taskStore.js` 导出 `q_`、`J_`，不需要再包一层 `localTool.tasks.save()`）
- ❌ 创建桶文件（两个文件不需要 `index.js` 重导出）
- ❌ 为了一致性把 3 行函数单独建文件
- ❌ 给同一个东西起两个名字

### 2.2 改名策略（切和改分离）

**剪切时不改名**。把代码从 App.js 原样搬到新文件，保证行为不变。这一步只做一件事：改变代码的物理位置。

**切完后，逐个评估是否可以改名**。评估标准：
1. 100% 确定语义（`func-mapping.txt` + grep 函数体坐实）
2. 引用范围可控（只在新文件和 App.js 的 import 中出现，没有动态字符串引用）
3. 改后认知负担明显降低（`saveTask` 比 `J_` 更容易理解）

满足全部三个条件 → 改。有一条不满足 → 不改。

### 2.3 通用铁律

- 每步 `npm run build` 必须通过
- 从 App.js 中删除已切出的代码，不做两份保留
- 小步提交
- 不碰 `entry.js`、`vendor/`、`dist/`、`.css`

## 5. 阶段二：切出状态管理

### 5.1 editorStore ⏸️ 暂停（2 轮尝试均失败，见翻车记录 #6、#7）

**实情**：`$` 并非本计划设想的"React Flow 节点/边 Zustand store"。经过 grep 定位和完整阅读，实际代码是：

- `$` = `var $ = Xt(...)` @ L19565-L20343，共计 **778 行**（不是本计划估计的 5000 行）
- `$` 是 **3D Director 场景编辑器** 的 Zustand store（selectObject、addCameraShot、addCrowdCharacters 等），不是 React Flow 画布状态
- `W`、`G`（setNodes、setEdges）在 `vendor.js` 中，与 `$` 无关
- `$` 依赖 App.js 内 30+ 个 helper 函数（`Tu`、`Ou`、`Eu` 等），这些 helper 又依赖更多内部常量

**为什么暂停**：

| 尝试 | 方案 | 失败原因 |
|------|------|----------|
| 第 1 轮 | editorStore.js import App.js 的 helper | 循环依赖，运行时 TDZ 错误 `Cannot access 'Lj' before initialization` |
| 第 2 轮 | 完整块迁移 2094 行（常量+helpers+store），零 App.js 依赖 | vendor.js 内部 minified 变量初始化顺序被打乱（`Pj`、`Aj` TDZ 错误），React.useContext 报 null |

**教训**：
- 不能从新文件 import App.js（App.js 只有 `export default`，无命名导出）
- 不能添加 `export { ... }` 到 App.js 然后再在新文件中 import（循环依赖必然导致 TDZ）
- 全块迁移（2094 行）会改变模块图，打乱 vendor.js 内部 `let`/`const` 的初始化顺序
- 依赖注入方案（工厂函数 + 30 个参数）可行但违反奥卡姆剃刀，收益不足以抵消新增复杂度
- **根本原因**：store 和它的 30+ 个 helper 函数是一体的，紧密耦合在 App.js 的模块闭包中。拆它们等于拆一个不可分割的有机体

**结论**：当前不拆。`$` store（778 行）及其 helper 函数保留在 App.js 中。如后续有需求，可用注释标记边界降低查找成本。

#### 6.1a. nodeTypes.js 提取方法

源：`lg` (L29005-29033) + `ug` (L29034-29036)。

**方法**：factory 函数。新文件 export `createNodeTypes({ Eh, li, Ya, ... })` 和 `createEdgeTypes({ jh })`。App.js 在所有节点组件变量声明后调用，赋值给 `lg` / `ug`。

**为什么不是包装层**：`createNodeTypes` 就是 `lg` 的定义本身，只是物理位置变了。App.js 调用时传的是同一个闭包中的组件引用，返回的是同一个对象。

#### 6.1b. ShortcutsModal.js 提取方法

源：L35174-L35421（约 250 行 `X.jsx` 内联 JSX）。

**方法**：标准 vendor 导入模板 + props 接口。

1. 新文件 import `{ i as e }`(rolldown-runtime) + `{ Nr as le, Ar as o, Mr as ae }`(vendor) → `var X = o()`
2. 组件接口：`{ isOpen, onClose }` — 原 `Ie &&` 条件转为 `if (!isOpen) return null`，原 `Le(false)` 转为 `onClose()`
3. 关闭按钮图标：原 `X.jsx(yn, { size: 16 })`(vendor 的 `yn`) → 内联 SVG X 图标，消除 vendor 图标依赖

**为什么跳过 useShortcuts 命名**：快捷键"功能"由 vendor 控件的 `Ie`/`Le` 状态驱动，不是独立 hook。提取的是快捷键"展示界面"而非快捷键"逻辑"。

5. **改名风险** — 切时不改，切完后逐个评估。只改引用范围可控的
6. **⚠️ 不能从新文件 import App.js** — App.js 只有一个 `export default`，没有命名导出。尝试添加 `export { ... }` 再 import 必然产生循环依赖，导致 vendor.js 内部 `let`/`const` 变量的 TDZ 错误
7. **⚠️ 全块迁移会打乱 vendor 初始化** — 移动大块代码（2094 行）到新文件，改变了 ES module 的加载图，vendor.js 中 minified 变量（如 `Lj`、`Pj`、`Aj`）的 `let`/`const` 初始化顺序被打乱，运行时抛 TDZ
8. **⚠️ 拆分前必须问："新文件能零 App.js 依赖吗？"** — 如果不能，拆分会引入循环依赖。这是判断是否可拆的最简单试金石
9. **⚠️ 6.1 教训：NodeAddMenu / FlowCanvas 不可拆** — `Z`(addNodeWithData) 闭包依赖 30+ 个局部变量，FlowCanvas 是组件 return 语句核心。这二者直到 node component 全部提取（6.3）前都不具备提取条件。nodeTypes 用 factory 函数传入组件引用可行（非包装层，是定义移动到新文件）
10. **✅ 提取组件的标准 vendor 导入模板（X.jsx 模式）** — 所有从 App.js 切出的组件文件遵循相同模板：`import { i as e } from "../../vendor/rolldown-runtime.js"` + `import { Nr as le, Ar as o, Mr as ae } from "../../vendor/vendor.js"` → `var X = o()` → 用 `X.jsx`/`X.jsxs` 写 JSX。不要改用标准 JSX 语法，保持与已有文件一致（LogoIcon、ToastContainer 等均用此模式）
11. **✅ vendor 图标替换策略** — 新文件不 import vendor 的图标组件（如 `yn`），改用内联 SVG。好处：(a) 减少 vendor 导入面，避免未来 vendor 重组的连锁修改；(b) 内联 SVG 自文档化，不用翻 vendor.js 才知道图标长什么样
12. **✅ factory 函数的界定标准** — 什么时候 factory 不是包装层而是合法提取？答：当 factory 本身是"定义的物理位置"，不是"对已有功能的二次封装"。`createNodeTypes({ Eh, li, ... })` 返回的对象就是原来的 `lg`，factory 只是把定义从 App.js 搬到新文件的载体
13. **✅ 评估"是否可拆"的决策树** — (1) 目标代码段内有没有对 App.js 局部闭包变量的引用？有 → 不可拆，除非全部提取。(2) 有没有对 vendor 的依赖？有 → 可拆，用标准 vendor 导入模板。(3) 有没有对其他 App.js 子模块的依赖？有 → 可拆，import 对应子模块。(4) 满足以上条件后问：切完后调用方的查找路径有没有变短？没有 → 不拆
14. **⚠️ vendor 导入必须用导出名，非内部名** — vendor.js 的 export 语句格式为 `vendorInternal as exportName`。import 时匹配的是 `exportName`，不是 `vendorInternal`。例如 `du as Wn` 的正确导入是 `import { Wn }`（不是 `import { du }`）。排查方法：`grep -o 'export{[^}]*}' src/vendor/vendor.js | tr ',' '\n' | sort` 列出全部导出名
15. **⚠️ `Un` 双用模式** — vendor 导出 `Un`（图标组件）与本地 `var Un = ae()`（createPortal）名称冲突时，用 `import { Un as Ee }` 将 vendor 的 `Un` 别名到 `Ee`，本地 `var Un` 正常声明。由于 vendor `Un` 只作为 `Ee` 进入作用域，本地 `Un` 不冲突
16. **⚠️ 6.3 全节点不可拆 — 已撤销全部 3 次提交** — 提取 `si`(NodeHeader，48 行独立函数) 后 `npm run build` 通过但运行时断裂：Chrome 扩展打不开，`Bd is not a function`。根因：App.js 模块级函数间存在隐式初始化依赖链，从中抽取任何函数都会改变 ES module 加载图拓扑，导致后续函数（`Bd` 等）加载失败。与 5.1 editorStore 翻车本质相同（全块迁移打乱初始化顺序）。**新增铁律：被多个其他模块级函数高频依赖的共享函数不可从 App.js 中抽取。** node 组件全保留在 App.js 内，仅通过 `lg = createNodeTypes(...)` factory 注册表实现逻辑分组
