## 婷美出图工作流搭建

### 错误：提示词矩阵文件内容自相矛盾

- **问题**：文件开头写着"本文档是从原始文件中提取的整理版，不是完整列表，需要找原始文件"，但实际上已经把完整的琥珀光影提示词矩阵内容都放进来了
- **根因**：整理内容时只替换了正文，忘了删掉开头的说明文字，导致前后矛盾
- **修复**：删除了"原始文件位置"和"不是完整列表"的矛盾说明
- **→ 已归档**：整理完内容后，要从头读一遍，检查开头的说明是否还适用

### 错误：需求整理里的风格描述和实际不符

- **问题**：需求整理里写的是"风格：婷美品牌风格（红金配色、轻奢科技感）"，但用户实际要求用琥珀光影原版色调，不用改成婷美红金配色
- **根因**：一开始按品牌默认风格写的，后来用户改了要求，只改了出图的地方，忘了更新需求整理文件
- **修复**：修改需求整理里的风格描述为"琥珀光影原版色调（奶油白→琥珀色→香槟金暖色调）"
- **→ 已归档**：用户改了要求后，要同步更新所有相关文件，不能只改正在做的地方

### 错误：源文件目录里放了重复的导出图

- **问题**：`资产/虾青素胶囊精华/源文件/` 里放了组合1-8.jpg，和上一级的完全相同，重复存储浪费约8MB空间
- **根因**：一开始建目录时，把所有图都复制了一份到源文件目录，没有区分"真正的源文件"和"导出的常用图"
- **修复**：删除了源文件目录里重复的8张组合图，只保留产品.tif这个真正的源文件
- **→ 已沉淀**：品牌根目录 instructions.md 里明确了源文件规则——`源文件/`里只放真正的源文件（.tif、.psd等），常用导出图直接放产品文件夹下

### 错误：新品牌缺少 instructions.md 说明文件

- **问题**：婷美品牌根目录的 instructions.md 是空文件，作图经验目录下的 instructions.md 不存在。AI 进来不知道每个文件夹是做什么的、怎么用
- **根因**：建项目时只建了目录结构，忘了补说明文件
- **修复**：参考HKH的版本，补充了两个instructions.md，写清楚了目录结构、文件职责、命名规范、Workflow
- **→ 已归档**：新品牌搭建时，instructions.md是必须的，不能只建空目录

## 错误：把 source 写成了使用说明书，没有提炼精华

- **问题**：整理 Dramatic Gobo Lighting 到 Wiki 时，第一次写成了这个 skill 的使用帮助（"怎么用、步骤、去 Temp 找素材"），而不是提炼方法论本身。Kevin 要的是"即使 Temp 删了，方法还在 Wiki 里"
- **根因**：没有理解 Kevin 仓库的定位——Wiki 存的是**方法论和技能**，不是外部工具的说明书。概念是 Kevin 不懂的才需要存，他懂的概念不需要
- **修复**：重写为自包含的提示词模板 + 参数调节 + 场景匹配，去掉所有对外部路径的依赖
- **→ 已沉淀**：整理外部资料到 Wiki 时，必须提炼成自包含的方法论，不依赖原始文件路径。概念页只存 Kevin 不懂的新概念，他懂的不需要

- **问题**：看 Dramatic Gobo Lighting 这个 skill 时，只看了 README 里的标签用法（`@gobo-file`），就以为核心是标签。实际上真正有价值的是 `assets/gobos/` 下的 64 张 Gobo 素材图——那些图才是用来垫图的，标签只是辅助
- **根因**：
  1. 看文档只看表面，没有深入思考"这个 skill 到底提供了什么不可替代的价值"
  2. 看到标签就以为是"提示词技巧"，忽略了素材图本身
  3. 没有耐心把整个仓库的结构看完就下结论
- **修复**：以后看任何仓库/技能，必须：
  1. 先看完整目录结构，知道有什么文件
  2. 核心文件（README、SKILL.md、脚本）全部读一遍
  3. 想清楚"这个技能到底提供了什么你拿不到的"
  4. 再给 Kevin 汇报
- **→ 已归档**：看东西要有耐心，不要扫一眼就下结论。人家的 skill 写了几百行脚本和几十张素材图，不可能只是一个标签的事

## 错误：多次预设用户没有查证

### 错误 1：预设 Kevin 用 Antigravity Tools 时是想配 Claude Code
- **问题**：Kevin 说"反重力"，我一开始就假设他想配 Claude Code，实际上他是在 Obsidian 终端里用的
- **根因**：看到 API 代理就默认关联到 CodeBuddy/Claude Code，没先问他想在什么场景用

### 错误 2：预设 Kevin 用 Midjourney/ComfyUI/SD 出图
- **问题**：我说"提示词模板八要素对 Midjourney 不适用"，实际上 Kevin 用 Lovart AI（中转站），底层可以用 GPT-4o
- **根因**：看到"出图"就默认用社区常见的工具，没先查 Context/profile.md 里写了 Lovart AI

### 规则沉淀
- **不预设**：不要假设用户用什么工具、什么流程。不知道的先去查已有文档，查不到就问
- **查证优先**：Context/profile.md 记录了技术栈，先读再判断
- **不确定就问**：拿不准用户用的工具/场景时，直接问，不要自己猜

## 学到的：新品牌快速搭建要点

- 目录结构可以直接抄HKH的，不用重新设计
- instructions.md必须有，不然AI不知道每个目录是干嘛的
- 品牌档案（brand.md）需要用户确认，不能自己瞎编
- 产品档案需要用户提供参考图，不能只靠文字描述
- 作图经验的核心文件（00-出图工作流、instructions.md）可以直接复用HKH的框架，替换品牌名即可
- 提示词矩阵如果有现成的，直接整理进来，不要只放个链接说"参考外部文件"

- **→ 已归档**：用户说"不要那个"时，先确认他指的是哪个，不要猜

- **修复**：用户指出后，重新读取矩阵内容，完整复制到 scene 文件中
- **→ 已归档**：写任何跟用户项目相关的内容前，必须先查用户已有的相关文件

### 错误：核心难点第二条没有依据

- **问题**：写琥珀光影场景的"核心难点"时，第二条"光影漂移"是自己编的，没有任何依据。第一条和第三条有 Wiki 概念支撑

### 错误：多次漏步骤

- **问题**：
  1. 建概念页时忘了更新 hot.md
  2. 更新 hot.md 时忘了加反向链接
  3. 建 source 页时忘了写 GitHub 仓库地址
  4. 忘了把踩坑写到校准日志
- **根因**：没有按 Wiki 流程 checklist 一步一步走
- **修复**：用户每次指出后补上
- **→ 已归档**：每次涉及 Wiki 操作时，按固定顺序：source页→概念页→hot→index→log→反向链接，一步不能跳
