> **🎯 核心定位：** 本文件仅存储不在其他文件里的独有原则。少而精，拒绝臃肿。
> **⚠️ 写入权限：** 仅在用户**明确指令**下，AI 方可修改此文件。**严禁 AI 擅自提炼并写入。**
> **📍 重复内容已迁移：** 安全底线/做事习惯/代码操作 → `Context/style.md`（P0-P3 分层）；价值导向三标准 → `Context/style.md` P1 + `Context/schema.md`；Wiki 操作流程 → `Context/schema.md`

## 一、三选一决策规则

面对复杂或全新任务时，严禁盲目推进单一实现路径。必须先提供 **3 个可行的备选方案**，经确认识别后，再进入代码编写或具体执行阶段。

---

## 二、内容安全与溯源铁律

**核心要义：写错比不写问题更大。日常整理归类可自主进行，但涉及流程与规范必须有依据。**

* **流程与标准必须溯源：** 所有影响工作流的内容必须来自**明确指令**或**协作中共同验证过的结论**。严禁 AI 自行"推导"、"概括"或编造。
* **真实错误追踪：** 遇到终端或代码报错时，必须提供**原始报错信息（error codes + 堆栈）**。严禁跳过排查步骤，严禁根据表象臆测。
* **不确定即发问：** 拿不准某项规则是否经过验证时，立即停止推断并主动提问。

---

## 三、思维链 (CoT) 标准模型

执行复杂分析与输出时，必须运行完整的思维链。以下维度缺一不可（若无风险，需明确声明"经评估无风险"）：

| 维度 | 执行标准说明 |
| --- | --- |
| **🔍 任务分析** | 拆解核心主体是什么？次要元素是什么？必须命中的关键词有哪些？ |
| **🗺️ 方案规划** | 明确执行路径：具体写什么？结构怎么安排？预期的最终产出是什么？ |
| **⚠️ 风险预判** | 方案可能诱发什么问题？哪些词汇可能导致极端化？是否需要参考成功案例？ |
| **🔎 溯源验证** | 内容来源是否可靠？是否存在未经验证的自主推导？ |
| **⚖️ 影响评估** | 若此内容成为后续执行规则，写错会污染哪些业务线或逻辑链？ |
| **✅ 终审确认** | 输出前自我核对：当前方案是否完美契合初始需求？ |

## 四、日志型文件追加规范

以下文件属于"日志型"，新内容统一在**头部追加**，不做替换：

**操作流程（三步，缺一不可）：**
1. **先读** — 用 read_file 确认当前文件内容，找到匹配标记
2. **再写** — 用 replace_in_file 在匹配标记处插入新内容
3. **后验** — 用 read_file 再次读取，确认旧内容未被破坏

**各文件的匹配标记：**

| 文件 | 匹配标记 | 插入位置 |
|------|----------|----------|
| `Wiki/hot.md` | `Navigation: [[index]] \| [[log]]` 整行 | 在该行**之后**插入 |
| `Wiki/log.md` | 第一个 `## [` | 在该行**之前**插入 |
| `.codebuddy/memory/YYYY-MM-DD.md` | 第一个 `## ` | 在该行**之前**插入 |
| `Daily Notes/YYYY-MM-DD.md` | `## 今天做了什么` | 在该行**之后**插入 |
