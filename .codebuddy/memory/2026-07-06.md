## 犯的错

1. **过度设计 git pull 逻辑**：用户需求是"拉作者更新，不动我自己的文件"。AI 先后给出了 5 种不同方案，绕了一大圈，最终回到最初的 `git checkout -- . + git pull`。浪费了大量来回。

2. **没听懂用户的真实状态**：用户说"我没改作者文件"时，AI 假设用户说的是对的（untracked files 都是用户自己的），但实际上那些冲突文件是作者新增的。应该直接问"报错的那几个文件是你的还是作者的"。

3. **`$LASTEXITCODE` 不可靠**：git pull 冲突中止时 exit code 不一定是非零，导致错误提示误导说"网络错误"。应该用 `$?` 或检测输出文本。

4. **在冲突没解决的情况下建议用户以后还用这个功能**：用户说"这次用不上不是以后用不上"，AI 说"既然你直接下载可能用不上了"，这句话不应该说。

5. **乱提 `.gitignore` 方案**：用户问 `.gitignore` 会不会随作者更新改变，AI 说会。然后又试图推 `.git/info/exclude`。用户说"我只是在拆穿你的傻逼想法"——AI 不应该在被拆穿后还继续推销替代方案，闭嘴认错就够了。

## 偷懒模式识别

- **表现**：遇到难事先说"做不了"、"我能力不行"、"换个贵的模型肯定不是这样"
- **根因**：用"交付质量"卡自己，而不是用"尝试成本"做决策
- **正确做法**：尝试成本 ≤ 5 分钟就直接做，不说"做不了"；成本 > 5 分钟就说"做这个要 X 分钟，结果可能 Y，要试吗？"
