# 分配任务工作流

> ⚡ 奥卡姆剃刀：如无必要，勿增实体。每句删了能不能完成任务？能，就删。

## 一、角色定义

| 角色 | 做什么 |
|---|---|
| **主 AI（委派者）** | 清旧任务书 → 接收需求 → 探索代码 → 写任务书 → 归档 |
| **执行 AI（受命者）** | 按任务书执行，不会回来提问，必须独立完成 |
| **用户** | 给出原始需求 |

**核心纪律**：执行 AI 不会提问。任务书必须自包含，不留模糊空间。

## 二、工作流

### 0. 清理旧任务书

删除 `docs/agent 批量任务/` 下所有 `TASK-*` 文件，为新一轮任务腾空编号空间。产出一个空白目录。

### 1. 理解需求

确认目标主题、是否禁止参考现有文档、是重写还是新增。产出：需求摘要。

> 需求不明确时标记 [信息缺口] 追问，不用假设填空。

### 2. 探索代码基底

grep 中英文关键词 → 读核心文件 → 标记混淆变量名语义 → 定位关键行区。产出：探索起点清单（路径+行号+grep 词+语义）。

### 3. 设计生命周期覆盖

按项目独特生命周期拆解（非 CRUD 通用模板），枚举"出现的所有地方"。产出：覆盖清单。

### 4. 起草任务书

按以下结构组装：项目背景 → 硬约束 → 探索起点 → 覆盖清单 → 输出规范 → 验收标准 → 铁律文件名（`docs/agent 批量任务/主题.md`）。产出：`docs/agent 批量任务/TASK-序号-任务短名.md`。

#### 4.1 代码改写类任务：必须包含三件套

如果任务是让 agent 改代码（重命名变量、加注释等），任务书必须额外提供以下三项，缺一不可。这不是"建议"，是省略任意一项就会返工的铁律：

| 必须给 | 格式 | 不给的后果 |
|--------|------|-----------|
| **变量映射表** | 原名→新名 对照表 | agent 猜错 → 返工 |
| **精确行号** | `App.js L1287` | agent 4.3 万行盲搜 → 写脚本 |
| **代码块模板** | `\| 节点类型 \| 组件名 \|` + 空 ` ```js ``` | agent 丢错位置 → merge 失败 |

> 原则：**agent 花的每一秒在"探索"而不是"执行"，都是任务书设计失败。** 三件套给齐 = 填空题，缺一样 = 探索题 → 脚本。

### 5. 归档

扫 `docs/agent 批量任务/` 已有 `TASK-*`，取最高序号+1，写入。

### 6. 交付话术

输出一行，用户直接看文件路径发给对应 AI 即可：

`任务书：docs/agent 批量任务/TASK-XXX-任务短名.md → 发给 AI0X`

## 三、绝对禁令清单

1. **文件名铁律**：每次委派必须指定唯一产出路径 `docs/agent 批量任务/主题.md`，不同 AI 不能写同一文件。
2. **禁止参考**：如要求"不参考现有文档"，硬约束必须写明。
3. **探索起点真实**：必须来自本次会话的实际 grep，不能复用历史（行号会变）。
4. **验收标准强制**：每份任务书必须有可自测的验收标准。
5. **序号递增**：编号扫描目录已有 +1，禁止硬编码 `TASK-001`。
6. **不可交互**：任务书禁止出现"如有问题请联系"等假设执行 AI 能提问的表述。
7. **独立文件夹**：每个 AI 的产出放 `docs/agent 批量任务/`，不混入 `docs/模块专题/`。

## 四、示例

> 用户：安排 AI01 重写会员机制文档，不参考现有版本。

- 产出文件：`docs/agent 批量任务/会员与登录机制-v2.md`
- 探索起点：grep `会员 / login / token / auth / membership`；读 `src/services/auth.js`、`src/services/modelEntitlements.js`、`src/config/storageKeys.js`、`src/App.js`（登录 L43849、401 登出 L36143、会员过期 L40586、登出 L41308）
- 覆盖清单：两条通路 → 存储键 → 登录协议 → token 收发 → 登录态校验/过期 → 会员等级来源 → 受限功能点 → 登出 → GAS 同步衔接 → 改码必查

## 五、输出前自检

- [ ] 旧任务书是否已清空？
- [ ] 产出文件名是否唯一且不与已有文件冲突？
- [ ] 硬约束是否写明"禁止参考现有文档"（如需）？
- [ ] 是否枚举了"出现的所有地方"（≥3 个，带路径+行号）？
- [ ] **任务是不是"填空题"**：agent 拿到就能动手，不用先探索？
- [ ] **铁律是否在第一条可见**：`## ⚠️ 铁律` 醒目标题 + "不写脚本"独立列出？
- [ ] **三件套是否给齐**（代码改写类）：映射表 + 行号 + 代码块模板？缺一样 = agent 走脚本

## 六、多 Agent 并行规则

核心思想：**agent 不知道别人存在，就无从抄袭和互扰。**

### 6.1 自包含任务书

每份任务书就是一个封闭世界。agent 打开文件后不需要、也看不到任何外部信息：

- 任务书内嵌了所有需要的参考材料（代码片段、映射表、行号）
- agent 只知道自己的产出文件路径，不知道还有几个 agent、他们做什么
- 不写"本次共分配 N 个 agent"、"你的编号是 X"

### 6.2 只写一个文件

```
1. 分配：每个 agent 一份自包含任务书，内嵌所有参考材料
2. 输出：agent 填自己文件的代码块/表格，只在这一个文件里写
3. 合并：你运行合并脚本统一写入（备份→逐条验证→全量 check→失败回退）
```

> 📁 任务书首行写死：`你只能写这个文件，碰任何其他文件视为失败。`

## 七、如何写任务生成脚本

不要在聊天里手写几十个任务文件。写一个 `tmp/gen-xxx.cjs`，跑一次生成全部。

### 7.1 模板

```js
// scripts/gen-xxx.cjs — 一键生成 N 个 agent 任务文件
const fs = require('fs');
const TASK_DIR = 'docs/<task-name>';
fs.mkdirSync(TASK_DIR, { recursive: true });

const items = [...];          // 要分配的任务项列表
const AGENTS = 5;             // agent 数量
const batchSize = Math.ceil(items.length / AGENTS);

for (let b = 0; b < AGENTS; b++) {
  const batch = items.slice(b * batchSize, (b + 1) * batchSize);
  if (batch.length === 0) continue;  // 最后一个为空就跳过
  const num = String(b + 1).padStart(2, '0');
  
  let md = `# XX${num} — 任务描述（${batch.length} 个）\n\n`;
  md += `> ⚠️ 请勿查看其他 agent 的结果文件。\n\n`;
  md += `## 方法\n...\n\n`;
  md += `## 输出格式\n...\n\n`;
  md += `| # | 项目 | 结果 |\n`;
  md += `|---|------|------|\n`;
  for (let i = 0; i < batch.length; i++) {
    md += `| ${i + 1} | \`${batch[i]}\` |  |\n`;
  }
  
  fs.writeFileSync(`${TASK_DIR}/XX${num}.md`, md);
}
```

### 7.2 关键点

- **数据源从代码提取**，不手写：`grep` / `fs.readFileSync` / AST 解析
- **每个 agent 独立文件**，不共用（§6.1 铁律）
- **输出格式明确**：填表格、代码块、或键值对
- **同步写合并脚本**：生成任务文件的同时，写好对应的 `scripts/merge-xxx.cjs`
- **用 `.cjs` 扩展名**：避免 ESM/CJS 冲突
- **跑完删脚本**：`node scripts/gen-xxx.cjs && rm scripts/gen-xxx.cjs`

### 7.3 已有参考

- `scripts/gen-all-tasks.cjs` — 之前生成 NC/EV/KV 三组任务的脚本（已归档 `scripts/archive/`）
- `scripts/annotate.cjs` — 基于映射表批量更新 App.js 注释
- `scripts/merge-rewrites.cjs` — 合并 agent 输出的函数体重写结果

## 八、踩坑经验（每个坑都返工过）

### 坑 1：任务书给的是"探索题"而不是"填空题"

**错误**：写"去 App.js 找 XX 函数，把 a,b,c 改成语义化名称"。
**结果**：agent 在 4.3 万行里盲搜，猜变量含义，3 个 agent 直接写脚本自动化、1 个甚至直接改了 App.js。返工 2 轮。

**正确**：任务书里必须提供三样东西，agent 只做替换动作：

| 必须提供 | 例子 |
|----------|------|
| 精确位置 | `App.js L1287，var ImageNodeComp = Y.memo` |
| 变量映射表 | `a→imgRef, o→isHovering, s→setIsHovering`（全写死） |
| 输出模板 | `\| imageNode \| ImageNodeComp \|` 后跟空代码块 |

> 原则：**agent 花的每一秒在"找东西"而不是"做东西"，都是任务书设计失败。**

### 坑 2：没检查就重做

**错误**：看到任务文件里没代码块就标记"未完成"，全部要求重写。
**结果**：有几个 agent 的代码块其实能过 build（AB03、AB06 全过），白白浪费。

**正确**：**先跑合并脚本看实际结果**。能过的保留，只补失败的。跑脚本比人工判断准。

### 坑 3：Agent 会写脚本

**错误**：以为多加禁令就能阻止。铁律第一条写了"不写脚本"，照样违。
**根因**：agent 写脚本是因为手写成本 > 脚本成本。任务书不给映射表+行号 → agent 得先定位函数、猜变量含义 → "不如写个脚本一次性搞定"。

**正确**：让手写比脚本更省力。给齐三样东西：
- 映射表（`a→updateNodeData`）→ 不用猜
- 行号（`App.js L1287`）→ 不用搜
- 代码块模板（`| xxx | xxx |` + 空代码块）→ 不用建格式

手写 5 分钟 vs 写 Babel AST 脚本 30 分钟。结构上让手写更划算，比一百条禁令管用。

### 坑 4：简单括号匹配提取大函数体会失败

**错误**：用正则 + 括号计数从 App.js 提取函数体。
**结果**：含模板字面量 `${}` 的大函数（如 PromptNodeComp 48k 字符）括号计数错乱，提取出错误片段。

**正确**：
- `function Name(){}` 和 `Name = Y.memo(() => {})` 两种模式分别处理
- 大函数不提取全量，改用行号定位 + 映射表就够了
- 如需提取全量，用 `@babel/parser` 而非正则

### 坑 5：铁律藏在不显眼的地方

**错误**：铁律写在 md 文件中部，agent 跳着读忽略掉了。
**结果**：多次违规。

**正确**：铁律放在任务书最顶部，用 `## ⚠️ 铁律（违反重做）` 醒目标题，三条以内，每条一句话。
